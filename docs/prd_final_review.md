## 一、整体评价与评分

### 总体评分：9.0 / 10（上一轮 7.5 / 10）

PRD v1.1 在 v1.0 基础上进行了**全面且高质量的修订**，第一轮 Review 中提出的 5 个 Blocker 和 8 个 Important 问题**全部得到有效解决**。修订内容体现了产品经理对技术约束的深入理解和对业务规则精确性的追求。

**核心改进亮点：**
1. 秒杀系统专项要求（5.4节）从性能指标、扩容预埋、用户体验三个维度给出了完整方案，7大预埋点覆盖从前端到基础设施全链路

4. 连定优惠规则精确到每天独立计价后各自打折，并明确了优惠互斥规则（最优价原则）
5. 数据模型采用"基础表+扩展表"模式，库存粒度按品类差异化定义，体现了良好的架构可扩展性
6. 新增消息通知体系、微信小程序平台约束、库存状态流转等关键章节，补全了功能和非功能需求的盲区

**仍可优化的方向（非Blocker，可在架构设计阶段迭代）：**

## 四、新发现的问题（如有）

经过对 PRD v1.1 的全面审查，以下是新发现的问题。**均为 Nice-to-have 或可在架构设计阶段解决的问题，无新的 Blocker**。

### 新问题1: 年卡会员"连续预定5天需中断2天"规则的算法精确性

- **严重程度**：Nice-to-have
- **问题描述**：
  - 3.2.1节预定频率限制中定义了"连续预定5天后需中断2天后才可继续预定"，但"连续"的精确判定逻辑未完全明确：
    - 例如：用户周一预定，周三预定，周五预定——这算连续3天还是3次？
    - 退票后是否恢复额度？
    - 建议采用"滑动窗口"模型（如"任意连续7天内最多预定5天"）会更清晰
  - 第一轮 Review（技术可行性问题4）也提到了此问题，但PRD保持了原始表述
- **修改建议**：
  - 这不是Blocker，因为业务意图已经清晰（防止年卡会员过度占用资源）
  - 建议在架构设计阶段与产品经理确认精确算法，技术方案文档中给出计算示例
  - 推荐算法："按预定日期的自然日连续性判定，连续入住5个自然日后，下一次预定的入住日期需≥第5天之后2个自然日"

### 新问题2: 多日票退款的日期粒度

- **严重程度**：Nice-to-have
- **问题描述**：
  - 3.7节退款规则明确"部分退票（如3天预定只退第3天）作为P2功能预留"
  - 但在P0阶段，如果用户预定了3天只想退最后1天，只能全额退全部3天的票——这可能导致用户体验问题和客服压力
- **修改建议**：
  - P0阶段可在客服层面支持手动处理部分退票（管理员后台操作）
  - 架构设计时`OrderItem`按天拆分存储，为P2的部分退票功能预留数据基础
  - 不影响当前PRD，在架构设计阶段预埋即可

### 新问题3: 动态定价中"自定义日期类型"的优先级规则

- **严重程度**：Nice-to-have
- **问题描述**：
  - 3.4节后台管理提到"日期类型管理：管理员可自定义日期类型"
  - 7.1节`PricingRule`和`DateTypeConfig`实体也有定义
  - 但如果管理员将某个日期同时设为"自定义类型"又配置了"特定日期特殊价"，两者之间的优先级关系未明确
- **修改建议**：
  - 建议在架构设计阶段明确定价优先级链：`特定日期特殊价 > 自定义日期类型价 > 系统日期类型价(工作日/周末/节假日)`
  - 不影响当前PRD，在技术方案中定义即可

### 新问题4: 购物车拆单后的支付体验

- **严重程度**：Nice-to-have
- **问题描述**：
  - 3.5.5节购物车定义了"系统自动拆分为多个子订单，各自独立走收入确认流程"
  - 但用户视角的支付体验未明确：拆分后是一次支付还是多次支付？
  - 如果是一次支付多个子订单，退款时部分退款的逻辑会比较复杂
- **修改建议**：
  - 建议架构设计阶段采用"父订单+子订单"模型：用户看到一个总价一次支付，后端拆分为多个子订单各自独立流转
  - 退款按子订单粒度操作
  - 不影响当前PRD，在架构设计阶段细化

### 新问题5: 缺少管理员操作的二次确认机制

- **严重程度**：Nice-to-have
- **问题描述**：
  - 后台管理中涉及高危操作（如批量下架商品、退款审批、会员权限变更、库存调整等），PRD虽然在3.4节增加了"操作日志"模块，但未定义操作前的二次确认机制
  - 第一轮 Review 附录的异常场景表也提到了"管理员误操作"风险
- **修改建议**：
  - 建议在架构设计阶段为高危操作增加确认弹窗
  - 可在后续迭代中增加"操作撤销"功能（如5分钟内可撤销）
  - 操作日志已有，审计追溯能力已具备
