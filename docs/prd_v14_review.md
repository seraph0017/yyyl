## 四、新发现的问题

### 问题1：次数卡有效期的计算起点未显式声明（⚠️ Minor）

- **问题描述**：
  - `TimesCardConfig` 有 `validity_days` 字段（如90天）
  - `TimesCard` 有 `start_date`、`end_date`、`activated_at` 字段
  - 但PRD未显式声明次数卡的有效期是从**激活日（activated_at）**开始计算还是从**其他日期**开始计算
  - 从字段设计推断应该是 `start_date = activated_at`，`end_date = activated_at + validity_days`，但这需要**显式声明**
  - 此问题在v1.3评审中已提出（变更6第5点），v1.4未修补
- **影响**：开发时可能出现理解偏差
- **建议**：在3.2.3节补充一行规则——"次数卡有效期从激活日（用户输入秘钥成功）开始计算：start_date = 激活日，end_date = 激活日 + validity_days"

### 问题2：多张次数卡的跨卡消耗规则缺失（⚠️ Minor）

- **问题描述**：
  - PRD明确"一个用户可同时持有多张次数卡"，且"预定时展示有效次数卡列表，用户选择用哪张卡"
  - 但如果用户选择的次数卡剩余次数不足以支付本次预定（如剩余2次，豪华营位需要3次），是否允许**跨卡使用**（从卡A扣2次+卡B扣1次）？
  - 当前PRD中未定义此场景
- **建议**：补充规则——"单次预定的次数消耗只能从**单张次数卡**扣减，不支持跨卡合并扣减。如选中的次数卡次数不足，提示'该次数卡剩余次数不足，请选择其他次数卡或联系客服'"（推荐不支持跨卡，以降低实现复杂度）

### 问题3：TimesCard实体的activation_code字段仍为字符串冗余（Informational）

- **问题描述**：
  - v1.3评审中指出 `TimesCard.activation_code` 字段冗余（应使用 `activation_code_id` 外键）
  - v1.4未修改，`TimesCard` 仍保留 `activation_code` 字符串字段
  - `ActivationCode` 已有 `used_by(user_id)` 可以反向关联到 TimesCard
- **影响**：不影响功能实现，但数据库设计不够规范化（冗余存储）
- **建议**：架构设计阶段的数据库ER图中，采用 `TimesCard.activation_code_id → ActivationCode.id` 外键替代字符串冗余

### 问题4：Web管理后台的RBAC权限模型粒度不足（⚠️ Minor）

- **问题描述**：
  - v1.3评审已指出权限模型仅有"管理员"和"员工"两个角色，v1.4未扩展
  - 3.4节权限管理表格仍然是两行：管理员（所有权限）、员工（验票+查看+部分商品管理）
  - Web管理后台引入了15个管理模块，仅靠"管理员/员工"两个角色无法满足精细化权限需求（如"财务人员"只看财务模块、"运营人员"管理商品和内容但不能看财务）
- **影响**：不阻塞P0/P1开发（初期团队小，管理员=老板），但后续营地扩展时将成为痛点
- **建议**：架构设计阶段预留RBAC模型（Role→Permission→Resource三层），PRD层面标注"管理员权限支持细分为子角色（P2功能预留）"即可

### 问题5：次数卡订单的total_amount值未定义（Informational）

- **问题描述**：
  - 3.2.3节定义"次数卡预定不收费，不走支付流程，直接生成已支付订单"
  - Order实体新增了 `payment_method = times_card`
  - 但 `total_amount` 的值未明确：是**0**（因为不收费），还是**该营位的市场价**（记录实际价值，便于统计）？
  - 上轮评审建议 `total_amount = 0`，v1.4未显式采纳或否决
- **影响**：影响财务报表的数据口径（如"总订单金额"是否包含次数卡订单）
- **建议**：架构设计阶段明确——`total_amount = 0` + 增加 `market_value` 字段（记录营位市场价，用于统计分析），或直接 `total_amount = 0` 且在统计报表中排除 `payment_method = times_card` 的订单
