## 三、新增问题

### N01. [HIGH] 编辑锁释放路径不一致
**位置**：第 11 节 `handleBeforeUnload`（L1032）

**现状**：
```typescript
// handleBeforeUnload 中
const url = `/api/v1/admin/cms/pages/${pageId}/lock/release`

// api/cms.ts 中 releaseCmsLock
return del(`/admin/cms/pages/${id}/lock`)
// 经 baseURL 拼接后实际为 /api/v1/admin/cms/pages/${id}/lock
```

**影响**：`handleBeforeUnload` 发送到 `/lock/release` 路径（对应后端不存在此路由），导致页面关闭时编辑锁无法释放，直到心跳超时（后端默认 5 分钟）才自动过期。

**修复建议**：
```typescript
const url = `/api/v1/admin/cms/pages/${pageId}/lock`
fetch(url, {
  method: 'DELETE',
  headers: { 'Authorization': `Bearer ${token}`, 'X-Site-Id': String(siteId) },
  keepalive: true,
}).catch(() => {})
```

### N02. [MEDIUM] readonly 模式对编辑器 UI 的影响未详细说明
**位置**：第 6.2 节 editor.vue、第 7.2-7.3 节 CanvasArea/PropertyPanel

**现状**：Store 已有 `readonly` 标志，但编辑器中如何使用该标志仅在第 11 节注释中提到「禁用拖拽/编辑/保存/发布」，未明确：
- CanvasArea 的 `<draggable>` 需设置 `:disabled="cmsStore.readonly"`
- PropertyPanel 表单元素需 `disabled` 状态
- 工具栏保存/发布/预览按钮需隐藏或禁用
- ComponentPanel 添加组件需禁用
- 自动保存定时器在 readonly 模式下不应启动

**建议**：在第 6.2 节数据流或第 7 节各组件设计中补充 readonly 模式的 UI 表现。
