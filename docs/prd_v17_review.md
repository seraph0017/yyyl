# 一月一露 — PRD v1.7 架构师评审报告

> **评审对象**：`prd/yyyl_prd_v1.7_increment.html` / `prd/yyyl_prd_v1.7_increment.md`
> **评审日期**：2026-06-18
> **评审角色**：架构师 Agent
> **评审结论**：CONDITIONAL APPROVED
> **加权总分**：8.62 / 10
> **状态说明**：可进入三端开发文档拆分，但开发文档必须吸收 HIGH 问题的约束；CRITICAL 问题为 0。

---

## 1. 总体结论

v1.7 PRD 已覆盖用户确认的 6 条需求，并且与现有 v1.6 CMS、订单、财务、退款、微信支付能力基本衔接。核心方向正确：

1. 小程序码统一走服务端生成与短码解析，规避微信 scene 长度限制。
2. 自定义页面二维码复用 v1.6 CMS `page_code`，避免新建重复页面系统。
3. 订单筛选与导出从同步列表能力升级为可审计导出任务，符合运营规模增长预期。
4. 资金从 `pending_amount` 到 `available_amount` 的结算流转与现有财务模型匹配。
5. 退款与取消订单解耦，补足 v1.5 退款增强未覆盖的真实运营场景。
6. 商家小程序版后台被降为 P2，不阻塞 P0 主链路，优先级合理。

本轮没有 CRITICAL 阻塞。存在 5 个 HIGH 问题，主要集中在资金并发一致性、退款状态机、二维码归因链路、导出权限模型和全额退款保留订单的风险边界。建议 PRD 可进入开发文档拆分，但三端开发文档必须把这些 HIGH 问题转成可执行技术约束和验收项。

## 3. HIGH 问题

### HIGH-01：资金结算需要明确并发锁、幂等键和负数保护的开发约束

**位置**：需求 4，订单资金结算账户流转。

**问题**：PRD 已提出“事务锁”“幂等”“账户不得为负”，但还没有明确开发文档必须采用的约束组合。支付回调、退款审批、订单完成结算、Celery 补偿可能并发操作同一 `FinanceAccount` 和同一 `Order`。如果只靠应用层判断，存在重复入账、重复结算、pending/available 扣成负数的风险。

**必须修补到开发文档**：

| 项目 | 要求 |
|------|------|
| 入账幂等 | 以 `payment_no` 或微信 `transaction_id` 建唯一约束；同一支付通知只能生成一笔 income/pending 流水 |
| 结算幂等 | `FinanceSettlement(order_id, settlement_no)` 或 `order_id + settlement_type` 唯一；`Order.settled_amount` 更新必须在同一事务内 |
| 行级锁 | 资金账户更新使用 `SELECT ... FOR UPDATE` 锁定 `FinanceAccount`；订单结算/退款同时锁定 `Order` |
| 非负约束 | 数据库层增加 `pending_amount >= 0`、`available_amount >= 0` 检查约束，或在迁移中给出等价保护 |
| 补偿任务 | Celery 只能处理 `settlement_status in ('unsettled','failed')` 且未完成幂等记录的订单 |

---

### HIGH-02：退款状态机需要收敛，避免 `cancelled/refunded/partial_refunded` 语义冲突

**位置**：需求 5，退款策略升级。

**问题**：PRD 同时出现订单状态 `refunded`、`cancelled`、`partial_refunded`，并提出“全额退款不取消订单”时订单仍可保持 `paid/verified/completed`。这符合业务诉求，但开发文档必须清晰拆分“服务状态”和“支付/退款状态”，否则前端筛选、电子票有效性、库存释放和财务报表会互相打架。

**必须修补到开发文档**：

| 项目 | 要求 |
|------|------|
| 状态分层 | `Order.status` 表示服务生命周期，`payment_status` 表示支付状态，新增/使用 `refund_status` 表示退款流程 |
| 保留订单 | `keep_order` 不应把 `Order.status` 改为 `refunded`；只更新 `payment_status/refund_status` 和退款记录 |
| 取消订单 | `cancel_order` 才允许更新 `Order.status=cancelled` 或指定终态，并作废电子票 |
| 全额退款保留订单 | 默认仅 super_admin 可操作，且电子票是否继续有效必须作为显式字段或强规则写入 |
| 前端筛选 | 订单筛选必须同时支持服务状态、支付状态、退款状态，避免一个“状态”字段承载所有语义 |

---

### HIGH-03：二维码渠道归因链路缺少从扫码到下单的持久化规则

**位置**：需求 1、需求 3。

**问题**：PRD 提到扫码日志、`Order.source_qrcode_id`、`source_channel`，但没有明确扫码后如何把二维码来源带到后续下单。如果只在解析时记录 ScanLog，用户跳转商品页后再下单，订单不一定能关联二维码渠道，导出中的 qrcode/channel 筛选会不准确。

**必须修补到开发文档**：

| 项目 | 要求 |
|------|------|
| 客户端存储 | 小程序扫码解析成功后，把 `qrcode_id/channel/scanned_at` 写入本地短期 attribution storage |
| TTL | 来源归因默认 TTL 建议 24 小时；同一用户重复扫码以后一次为准 |
| 下单传参 | `POST /api/v1/orders` 请求体或请求头携带来源；后端校验二维码仍有效后写入 Order |
| 未登录用户 | 未登录扫码也应保留本地来源，登录/下单时补写 |
| 导出字段 | 订单导出明确包含来源二维码标题、渠道、扫码时间或归因时间 |

---

### HIGH-04：订单导出文件的下载权限和生命周期需要更严格

**位置**：需求 3，订单高级筛选与导出。

**问题**：PRD 已提出导出文件保存 7 天、敏感字段脱敏、审计日志，但缺少下载 URL 的授权方式。若导出文件直接放在 `/images` 或可公开访问目录，可能绕过后台权限下载包含订单和手机号的数据。

**必须修补到开发文档**：

| 项目 | 要求 |
|------|------|
| 存储目录 | 导出文件不得放在公开静态目录；应放私有目录，通过认证接口流式下载 |
| 下载鉴权 | `GET /admin/orders/export-tasks/{id}/download` 校验当前 admin、site_id、导出权限 |
| 链接有效期 | 如生成临时下载 token，TTL 不超过 10 分钟，且绑定 admin_id/export_task_id |
| 敏感字段 | 完整手机号、openid、身份证等默认不导出；完整手机号需单独权限 |
| 清理任务 | Celery 清理过期文件同时更新任务状态或 `file_url=null` |

---

### HIGH-05：小程序码生成需补充微信接口限流、失败重试和路径白名单

**位置**：需求 1，小程序码统一生成与管理。

**问题**：PRD 说明了生成、复用、停用，但没有明确微信接口失败和非法路径的处理策略。二维码生成属于外部 API 调用，容易受 token 过期、微信限流、网络失败影响；路径解析如果不做白名单，会形成内部开放跳转风险。

**必须修补到开发文档**：

| 项目 | 要求 |
|------|------|
| AccessToken | 微信 access_token 必须缓存并在失效时刷新，不可每次生成都获取 |
| 限流重试 | 微信接口失败按错误码区分：限流/网络错误可重试，参数错误直接失败 |
| 白名单 | 解析返回 path 只能来自后端枚举模板，不接受数据库中的任意 path 直接跳转 |
| 图片校验 | 微信返回图片后校验 Content-Type 和文件大小，防止把错误 JSON 当图片保存 |
| 批量生成 | 若后续支持批量生成，必须走异步任务，避免后台请求超时 |

## 4. MEDIUM 建议

| 编号 | 建议 | 说明 |
|------|------|------|
| M01 | 将“活动”命名拆成 `activity_product` 与 `cms_activity_page` | 当前 `activity_page` 可理解，但开发接口中建议更明确，避免和 Product.type 混淆 |

## 7. 结论与下一步

评审结论：**CONDITIONAL APPROVED，8.62 / 10**。

v1.7 PRD 可以进入三端开发文档拆分阶段。拆分时必须把 5 个 HIGH 问题转写为后端、管理后台、小程序三端的开发约束：

1. 后端开发文档重点补资金锁、幂等、退款状态机、导出私有下载、小程序码微信接口封装。
2. 管理后台开发文档重点补订单多状态筛选、导出权限、退款高危确认、二维码管理状态展示。
3. 小程序开发文档重点补扫码分发页、归因本地存储、CMS 页面跳转、错误页。

若开发文档吸收上述 HIGH 问题，预计三端文档可达到 APPROVED 标准；若忽略 HIGH 问题，则资金与退款模块存在生产级风险。
