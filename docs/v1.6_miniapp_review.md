## CRITICAL 问题复审

### C01：API 响应格式与现有 request.ts 封装不一致

**状态**：✅ 已修复

**验证**：
1. 文档 7.2 节开头新增醒目的 `⚠️ 重要约定`，明确所有 CMS 接口统一使用 `{ code: 0, message: "success", data: {...} }` 格式

3. 明确注明 **不使用** PRD 中的 `{ success: true, data }` 格式，且标注"后端实现时须遵循此约定"
4. 变更记录中 `"ok"` 也统一修正为 `"success"`

**修复质量**：优秀。不仅修正了格式，还加了面向后端开发者的约束说明，可有效避免前后端理解分歧。

### C02：CmsRenderer 的 props 传递方式存在 Vue3 陷阱

**状态**：✅ 已修复

**验证**：
1. CmsRenderer 模板中已从 `:props="comp.props"` 改为 `:data="comp.props"`

3. 3.4 节统一接口约定中明确写为"接收 `data` prop（组件的配置数据）"
4. 子组件模板中引用均为 `data.xxx`，无 `props.props` 的双重嵌套问题

**修复质量**：优秀。全量修改，无遗漏。

## HIGH 问题复审

### H01：checkCmsPageVersion 接口定义了但未使用

**状态**：✅ 已修复

**验证**：
1. **首页（5.6 节）**：`loadPageConfig` 函数中，当有缓存时先调用 `checkCmsPageVersion(PAGE_CODE, cachedVersion.value)`，仅在 `has_update === true` 时才请求完整配置，`has_update === false` 时直接 `return`
2. **CMS 通用容器（6.1 节）**：相同逻辑，有缓存时先 check，无更新则跳过完整请求
3. 流程为：有缓存 → 渲染缓存 → check 接口 → 有更新 → 拉取完整配置 → 更新渲染+缓存；无更新 → 跳过
4. 无缓存时直接请求完整配置（不调 check），逻辑正确

**修复质量**：优秀。符合 PRD 设计 check 接口的初衷，大幅减少不必要的大 JSON 传输。

---

### H02：优惠券组件（CmsCoupon）缺少领券 API 的认证处理

**状态**：✅ 已修复

**验证**：
1. 3.4.7 节第 7 点新增「未登录处理」：检查 Pinia userStore 登录状态 → 未登录跳转 `/pages/login/index` → 登录成功返回后自动重试
2. 3.4.7 节第 8 点新增「领券失败状态 UI」，覆盖 5 种场景：
   - 已领取 → 灰色"已领取"
   - 已抢光 → 灰色"已抢光"
   - 未开始 → 灰色"未开始"
   - 已过期 → 灰色"已过期"
   - 其他失败 → toast 提示
3. 样式设计中补充了 `&__btn--disabled` 样式（opacity: 0.5, pointer-events: none）

**修复质量**：优秀。场景覆盖全面，UI 状态与交互逻辑清晰。

## MEDIUM 问题复审

### M01：缺少组件级错误边界

**状态**：✅ 已修复

**验证**：CmsRenderer 3.2 节 `<script setup>` 中添加 `onErrorCaptured` hook，捕获子组件渲染异常，`console.warn` 记录组件名和错误信息，返回 `false` 阻止冒泡。

**备注**：Vue3 的 `onErrorCaptured` 可以阻止错误传播，但对于渲染阶段的异常，被捕获的组件可能仍会显示空白或部分渲染。这在实际开发中可以通过 `v-if` + 组件状态控制进一步优化，但当前方案已满足"不影响其他组件"的基本要求。
