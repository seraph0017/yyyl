**修复质量：方案正确，但实现有执行阻断问题**

新增 §9.1 节完整描述了 `slowapi` 方案，包含 `main.py` 初始化、路由装饰器、自定义 key_func。`requirements.txt` 已补充 `slowapi`。PRD 要求的三种限流规则均已覆盖（100/分/IP、30/分/IP、60/分/用户）。

**问题 1（HIGH — 循环导入）**：§9.1 路由装饰器示例中 `from main import limiter`，而 `main.py` 中 `app.include_router(cms.router)` 会导入 `routers/cms.py`。这构成 **循环导入**，运行时会抛出 `ImportError`。

**修复方案**：将 `limiter` 实例移至独立模块，或通过 `request.app.state.limiter` 访问：

```python
# 方案 A：独立模块 utils/limiter.py
from slowapi import Limiter
from slowapi.util import get_remote_address
from config import settings

limiter = Limiter(key_func=get_remote_address, storage_uri=settings.REDIS_URL)

# main.py 中
from utils.limiter import limiter
app.state.limiter = limiter

# routers/cms.py 中
from utils.limiter import limiter
```

**问题 2（MEDIUM — key_func 取值方式）**：`get_admin_id()` 中 `request.state.admin` 假设 auth 中间件将 admin 注入 request.state，但项目实际使用 FastAPI `Depends(get_current_admin)` 依赖注入模式，admin 不在 request.state 上。

**修复方案**：
```python
def get_admin_id(request: Request) -> str:
    # 从 JWT token 解析 admin_id，与 get_current_admin 使用相同逻辑
    from middleware.auth import decode_token
    token = request.headers.get("Authorization", "").replace("Bearer ", "")
    payload = decode_token(token)
    return str(payload.get("sub", "anonymous"))
```

### M02: CmsPageVersion 索引缺少 DESC 排序 → ❌ 未正确修复

**修复意图正确，但代码存在语法错误，会导致 `NameError`。**

当前代码（第 153-154 行）：
```python
__table_args__ = (
    Index("idx_cms_pv_page", "page_id", version_number.desc()),
    Index("idx_cms_pv_published_at", published_at.desc()),
    ...
)
```

Python 类体按顺序执行，`__table_args__` 在第 151 行定义，但 `version_number` mapped_column 在第 162 行、`published_at` 在第 174 行才定义。此时 `version_number` 标识符尚不存在，**运行时会抛出 `NameError: name 'version_number' is not defined`**。

**修复方案**：使用 `text()` 表达式或列字符串：

```python
from sqlalchemy import text

__table_args__ = (
    Index("idx_cms_pv_page", "page_id", text("version_number DESC")),
    Index("idx_cms_pv_published_at", text("published_at DESC")),
    {"comment": "CMS 页面版本历史"},
)
```

### M06: `<a>` 标签 href 白名单校验未实现 → ❌ 未修复

`sanitize.py` 中仍仅有 `ALLOWED_URL_SCHEMES = {"http", "https"}`，未增加对 `/pages/` 相对路径前缀的白名单校验。nh3 的 `url_schemes` 参数只过滤绝对 URL 协议，不处理以 `/` 开头的相对路径。

PRD 1.3.8 要求：`<a>` 标签 `href` 仅允许 `http://`、`https://`、`/pages/` 开头的 URL。

**修复方案**（在 `sanitize_html()` 后增加后处理）：
```python
import re

def _filter_relative_hrefs(html: str) -> str:
    """过滤 <a href> 中不合规的相对路径"""
    def _check_href(match):
        href = match.group(1)
        if href.startswith(("http://", "https://", "/pages/")):
            return match.group(0)
        # 移除不合规的 href
        return match.group(0).replace(match.group(1), "#")
    return re.sub(r'href="([^"]*)"', _check_href, html)

def sanitize_html(html: str) -> str:
    cleaned = nh3.clean(html, ...)
    return _filter_relative_hrefs(cleaned)
```

### M07: 乐观锁比较逻辑不够严谨 → ❌ 未修复

`save_draft()` 第 1213 行仍为：
```python
if data.draft_updated_at and page.draft_updated_at:

```

当页面已有草稿（`page.draft_updated_at is not None`）但客户端未传 `draft_updated_at` 时，乐观锁完全不生效，并发覆盖风险依然存在。

**修复方案**：

if page.draft_updated_at is not None:

            status_code=409,
            detail={
                "code": "CMS_DRAFT_CONFLICT",
                "message": "该页面已有草稿，请先获取最新版本后再编辑（需传入 draft_updated_at）",
            },
        )
    if page.draft_updated_at > data.draft_updated_at:
        raise HTTPException(
            status_code=409,

## 本轮新发现问题

### N01（MEDIUM）: `delete_page` 错误码复用不当

**位置**：`services/cms_service.py` → `delete_page()` 第 1389-1391 行

**问题**：当页面类型非 custom 时拒绝删除，但错误码使用了 `CMS_PAGE_NOT_FOUND`（404）。这是业务规则拒绝，不是"未找到"，应使用 400 + 专用错误码。

**修复**：
```python
raise HTTPException(
    status_code=400,
    detail={"code": "CMS_PAGE_DELETE_NOT_ALLOWED", "message": "仅 custom 类型页面可删除"},
)
```

### N02（LOW）: 宣传页缓存 key 与页面缓存 key 冗余

**位置**：`_clear_page_cache()` 第 1093-1098 行

**问题**：宣传页实质上是 `page_code="admin_landing"` 的 CmsPage，其缓存已通过 `cms:page:{site_id}:admin_landing` 管理。额外维护 `cms:landing:{site_id}` key 会导致：
1. 两个 key 缓存同一份数据
2. `get_landing_page` 路由调用 `get_published_page(db, ..., page_code="admin_landing")`，实际使用的是 `cms:page:` key，`cms:landing:` key 无读取入口。

**建议**：移除 `cms:landing:{site_id}` 相关逻辑，或在 `get_landing_page` 路由中使用独立的缓存读写路径。
