## 四、技术完整性检查

### 4.1 数据模型

**覆盖度**：42张表，覆盖了PRD中所有业务实体及必要的扩展表。

| 检查项 | 状态 | 说明 |
|--------|:----:|------|
| 公共字段Mixin | ✅ | id/created_at/updated_at/is_deleted |
| site_id多营地预留 | ✅ | 核心表均含site_id DEFAULT 1 |
| 索引设计 | ✅ | 关键表都有索引定义 |
| JSONB字段使用合理 | ✅ | 用于灵活字段（custom_fields/privileges/images等） |
| 外键关系完整 | ✅ | ER图+表定义中FK清晰 |

**发现的问题**：

| # | 问题 | 建议 |
|---|------|------|
| P1 | **User表合并了PRD的Member实体**：`is_member`和`points_balance`直接放在User表。PRD定义了独立的`Member`实体。虽然功能等价，但如果后续有会员等级扩展（PRD已有`level`字段），User表将变得臃肿 | 可接受，但建议在注释中说明设计决策 |
| P2 | **Ticket表的order_item_id为NULL**：多日票场景下，每天生成的Ticket应关联到当天的OrderItem。但架构中order_item_id为`NULL`类型，未标注是必填还是选填 | 建议改为NOT NULL，或明确说明何种场景为NULL |
| P3 | **缺少DailyReport/WeeklyReport/MonthlyReport表定义**：目录结构9.1提到`stats.py`含这三张表，但2.3节未给出表结构定义 | 需补充表结构 |
| P4 | **AnnualCard缺少order_item_id**：年卡购买应通过OrderItem跟踪，但AnnualCard表只有order_id | 低优先级，order_id已可追溯 |
| P5 | **CartItem缺少cart_type字段**：PRD要求区分小商店/周边两种购物车类型（用于拆单），但CartItem中无此字段 | 建议在Cart或CartItem中增加cart_type，或通过product.type推断 |
| P6 | **FinanceTransaction表缺少user_id**：退款等操作需关联用户，仅有order_id可间接查到用户但查询不便 | 建议添加user_id字段 |

| A2 | **消息管理模块缺少B端API**：PRD要求"通知模板配置、消息发送记录"管理功能，当前仅有C端通知查询(#170-173)和客服信息配置(#151)，缺少通知模板管理和发送记录查询的管理端API | 新增3-4个管理端通知API |
