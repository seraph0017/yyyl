## 一、整体评价

### 1.1 总体评分：7.5 / 10

PRD整体质量较高，覆盖了露营地小程序的核心业务场景，7大品类划分清晰，用户旅程描述完整。以下是主要优点和不足：

**优点：**
- 7大品类的业务规则描述清晰，每个品类都有独立的业务规则表、异常处理和验收标准
- 用户旅程覆盖了5条核心路径，页面跳转关系图直观
- 会员系统的"连续预定5天需中断2天"等防刷规则考虑周全
- 收入管理的三种流转模式定义合理，符合资金安全要求
- 免责声明的法律效力设计完整（签署时间+微信ID+身份信息）

**需要改进的关键问题：**
1. **秒杀扩容设计严重不足**——500 QPS的性能指标远远不够，且缺乏扩容预埋设计
2. **7大品类统一数据模型的可行性存疑**——Product表用JSON字段兼容7种差异极大的品类，后期维护代价大
3. **多项业务规则存在模糊地带**——连定优惠计算、押金部分扣除、积分兑换等缺乏精确定义
4. **缺失多个关键功能模块**——消息通知、客服系统、分享裂变、评价体系等
5. **合规设计不够深入**——缺少个人信息保护法(PIPL)合规细节

## 二、业务完整性问题

### 问题1：装备租赁的押金部分扣除规则不明确

- **位置**：PRD 第3.1.3节 装备租赁 → 异常处理
- **问题描述**：PRD仅提到"管理员可部分扣除押金后退款"，但缺少以下关键定义：
  - 扣除比例/金额的计算依据是什么？（固定金额？按损坏程度评估？）
  - 是否需要用户确认/申诉？还是管理员单方面决定？
  - 扣除后的款项如何入账？进入哪个账户？
  - 是否需要拍照留证？损坏记录如何存档？
- **修改建议**：
  1. 增加"损坏评估流程"：管理员拍照→选择损坏等级（轻微/中度/严重/报废）→系统计算扣除金额→通知用户→用户可申诉→最终退款
  2. 明确扣除金额的计算规则，建议提供可配置的损坏赔偿比例表
  3. 扣除的押金应记入"设备维护收入"独立科目

### 问题2：连定优惠的计算逻辑存在歧义

- **位置**：PRD 第3.1.1节 日常露营票 → 业务规则 → 连定优惠
- **问题描述**：
  - "连续预定2天打8折"——8折是对第2天打折还是2天都打折？
  - "同一订单超5人打8折"——是从第6人开始打折，还是超过5人后全部人打折？
  - 连定优惠和节假日定价是否叠加？（如连续预定2天，第1天是工作日128元，第2天是周末168元，折扣基准是什么？）
  - 年卡会员享受免费预定的营位是否还适用连定优惠（免费×0.8=免费，语义上冗余）？
  - 多种优惠是否互斥？（连定优惠 vs 会员折扣 vs 积分抵扣，优先级是什么？）
- **修改建议**：
  1. 明确折扣作用范围：建议"连续预定N天，全部天数享受X折"
  2. 明确人数折扣的起算方式：建议"同一订单人数≥5人时，全部人享受X折"
  3. 增加优惠互斥/叠加规则说明：建议采用"最优价格原则"——系统自动计算各种优惠方案，取用户最优价
  4. 列出明确的计算示例

### 问题3：普通会员积分体系不完整

- **位置**：PRD 第3.2.2节 普通会员
- **问题描述**：
  - 积分有效期是多长？永久有效还是年度清零？
  - 退票/退款时积分是否回收？
  - "限时活动免费预定"需要多少积分？兑换比例是什么？
  - "限时商品折扣或兑换"的规则是什么？管理员如何配置积分兑换活动？
  - 积分是否有等级体系？（如铜/银/金/钻石会员）
  - 年卡会员消费是否也积累积分？（年卡会员和普通会员的关系是包含还是互斥？）
- **修改建议**：
  1. 补充积分有效期规则（建议：自获取之日起12个月有效）
  2. 补充退款积分回收规则
  3. 补充积分兑换的配置逻辑：管理员可创建"积分活动"，设置兑换比例和库存
  4. 明确年卡会员和普通会员的关系——建议：年卡会员同时也是普通会员，消费仍累积积分

### 问题4：收入管理缺少多个流转场景

- **位置**：PRD 第3.3节 收入管理
- **问题描述**：当前只定义了3种收入确认模式，但遗漏了以下场景：
  1. **装备租赁收入**：租赁费何时确认？是验票拿装备时确认还是归还后确认？
  2. **押金的资金流转**：押金在哪个账户？部分扣除时如何处理？
  3. **日常活动/特定活动预定**：活动票的收入确认时机是什么？是验票参加活动时确认？
  4. **年卡会员费**：年卡费用是即时确认还是按月分摊？
  5. **退款的资金流转**：从哪个账户扣除？如果已经从待确认转到可提现后用户申请退款怎么办？
  6. **积分兑换订单**：全额积分抵扣的订单没有资金流转，但库存需要扣减
- **修改建议**：
  补充完整的收入确认矩阵：

  | 品类 | 收入确认时机 | 退款规则 |
  |------|-------------|----------|
  | 日常/活动露营票 | 验票入营时 | 验票前可退→从待确认扣除；验票后退款→从可提现扣除 |
  | 装备租赁-租赁费 | 验票拿装备时 | 拿装备前可退→从待确认扣除 |
  | 装备租赁-押金 | 单独核算，不进入收入 | 归还后退押金 / 损坏扣除后退剩余 |
  | 日常/特定活动票 | 验票参加时 | 活动前可退 |
  | 小商店 | 支付即确认 | 联系客服退款→从可提现扣除 |
  | 周边商品 | 确认收货时 | 收货前可退→从待确认扣除 |
  | 年卡会员费 | 支付即确认 | 购买后X天内可无条件退款（需明确） |

### 问题5：后台管理功能缺少关键模块

- **位置**：PRD 第3.4节 后台管理
- **问题描述**：
  1. **缺少"营地日历"管理视图**：露营地运营核心是按日期管理，需要一个日历视图来直观展示每天各营位的预定情况、库存余量、价格类型（工作日/周末/节假日）
  2. **缺少"日期类型管理"功能**：PRD提到"管理员可自定义日期的类型"，但后台模块中没有对应的管理入口
  3. **缺少"消息/通知管理"**：无法向用户推送通知（如活动上新、天气预警、验票提醒）
  4. **缺少"客服管理"**：小商店提到"一键联系客服"，但未定义客服系统的后台管理
  5. **缺少"退票审批队列"**：订单管理中提到退票审批，但没有独立的审批工作流定义
  6. **缺少"操作日志"**：关键操作（改价、退款、权限变更等）需要审计日志
- **修改建议**：在后台管理模块中增加以上6项功能

### 问题6：缺少消息通知体系

- **位置**：PRD 全文缺失此模块
- **问题描述**：当前PRD完全没有定义消息通知体系，但以下场景强依赖通知：
  1. 定时开票提醒："您关注的XX活动将于明天20:00开票"
  2. 订单状态变更通知："您的订单已确认/已取消/已退款"
  3. 验票提醒："您预定的营位就在明天，别忘了出发！"
  4. 活动上新通知："新活动上线，快来看看！"
  5. 押金退还通知："您的押金XXX元已退还"
  6. 支付超时提醒："您的订单将在15分钟后取消，请尽快支付"
  7. 周边商品发货通知："您的订单已发货，快递单号：XXX"
  8. 年卡到期提醒："您的年卡将于XX天后到期，续费享受XX优惠"
- **修改建议**：
  1. 增加"消息通知"独立章节
  2. 通知渠道：微信服务通知（模板消息/订阅消息）、小程序内消息中心
  3. 注意：微信小程序订阅消息需要用户主动订阅，PRD需定义用户订阅引导策略

### 问题7：缺少商品分享与社交裂变功能

- **位置**：PRD 全文缺失
- **问题描述**：微信小程序天然具备社交分享能力，但PRD未涉及任何分享功能：
  - 商品/活动分享到微信好友或朋友圈
  - 分享海报生成
  - 拼团/邀请有礼等裂变玩法
  - 分享带参二维码（追踪分享来源）
- **修改建议**：至少在P1中增加"商品/活动分享"功能（分享给好友+生成海报），更高级的裂变玩法可放P2

### 问题8：缺少评价/点评系统

- **位置**：PRD 全文缺失
- **问题描述**：
  - 用户露营后无法评价营位体验
  - 用户无法对租赁装备的质量进行反馈
  - 活动结束后无法评价活动质量
  - 周边商品无法评价
  - 缺乏用户口碑积累，不利于新用户决策
- **修改建议**：在P2中增加"评价系统"模块，至少支持星级评分+文字+图片评价

## 三、技术可行性问题

### 问题1：7大品类统一Product模型的可行性风险

- **位置**：PRD 第7.1节 数据字典
- **问题描述**：当前设计将7种差异极大的品类统一放入Product表，差异化属性用`attributes(JSON)`字段承载。技术风险如下：
  1. **查询性能**：JSON字段无法高效索引，按营位属性（有电/无电）筛选需要全表扫描或额外索引策略
  2. **数据完整性**：JSON字段无法做数据库层面的约束校验，数据质量依赖应用层
  3. **业务差异过大**：
     - 露营票需要：日期选择、营位属性、动态定价、连定优惠
     - 装备租赁需要：押金、归还流程、损坏评估
     - 小商店商品需要：购物车、即时购买
     - 周边商品需要：收货地址、物流跟踪、SKU规格（尺码/颜色）
  4. **代码复杂度**：统一模型会导致Service层大量的if-else分支，可维护性差
- **修改建议**：
  PRD层面建议明确以下策略（由技术架构师在架构设计阶段最终决策）：
  - 方案A（推荐）：Product基础表 + 各品类扩展表（如 `camping_ticket_ext`、`rental_ext`、`shop_product_ext`），兼顾统一管理和品类差异
  - 方案B：完全独立的品类表，但共享订单和支付体系
  - 建议PRD在数据字典中补充各品类特有字段的说明，方便架构师做出合理决策

### 问题2：库存模型设计不够精细

- **位置**：PRD 第7.1节 Inventory实体
- **问题描述**：
  1. 当前Inventory表只有 `product_id, date, total, available, locked`，但不同品类的库存粒度完全不同：
     - 营位票：按日期+营位维度（孤品是1:1，自由营位是按人数上限）
     - 装备租赁：按日期+装备维度，且同一装备不同日期可被不同人租赁
     - 小商店/周边商品：不按日期，仅按商品维度，且需要支持入库/出库操作
     - 活动票：按场次时间段维度
  2. 缺少"锁定超时释放"的机制定义：用户下单锁定库存后，多久未支付自动释放？15分钟？30分钟？
  3. 缺少"预扣库存"和"实际扣减"的状态流转定义
- **修改建议**：
  1. 在PRD中补充库存的粒度定义和状态流转：`可用→锁定（下单）→已售（支付成功）/ 释放（支付超时/取消）`
  2. 明确支付超时释放时间（建议：秒杀场景5-10分钟，普通场景30分钟）
  3. 补充入库/出库操作的流程定义（针对小商店和周边商品）

### 问题3：动态定价规则缺乏精确定义

- **位置**：PRD 第3.1.1节 动态定价 + 第7.1节 PricingRule实体
- **问题描述**：
  1. `date_type` 只说了工作日/周末/节假日，但没有定义：
     - 多天预定时，每天价格独立计算还是取最高价？（如周五-周日连定3天，周五是工作日价、周六周日是周末价？还是全部按周末价？）
     - 是否支持"特殊日期特殊价"？（如某个特定日期单独设价，不走类型规则）
  2. PricingRule 的 `discount_rules(JSON)` 没有定义具体的数据结构
  3. 活动票的定价方式未说明——是固定价还是也支持动态定价？
- **修改建议**：
  1. 明确多天预定的计价规则：建议按"每天独立计价，再叠加连定优惠"
  2. 支持"日期级别"定价覆盖：管理员可对任意日期设置特殊价格，优先级高于日期类型规则
  3. 补充 discount_rules 的结构定义示例
  4. 活动票建议固定价格，不适用动态定价

### 问题4：年卡会员"连续预定5天需中断2天"规则的技术复杂度

- **位置**：PRD 第3.2.1节 年卡会员 → 预定频率限制
- **问题描述**：
  - "连续预定5天后需中断2天"的判定逻辑复杂：如何定义"连续"？是自然日连续还是预定记录连续？
  - 如果用户周一预定了周三、周五（不连续），算不算使用了2天额度？
  - 如果用户退票了其中某天，是否恢复连续预定额度？
  - 跨月/跨年如何计算？
  - 这个规则的检查是在前端还是后端？高并发下是否需要加锁？
- **修改建议**：
  1. 精确定义"连续"的含义：建议定义为"在任意连续7天内，最多预定5天"（滑动窗口模型）
  2. 明确退票后的额度恢复规则
  3. 提供1-2个完整的计算示例
  4. 这个规则应在后端严格校验，前端仅做提示

### 问题5：缺少SKU/规格体系定义

- **位置**：PRD 第3.1.7节 周边商品 + 第7.1节 数据字典
- **问题描述**：
  - 周边商品（衣服、帽子）通常需要SKU管理：尺码（S/M/L/XL）、颜色等
  - 当前Product和Inventory模型没有SKU维度，无法支持"红色M码库存3件，蓝色L码库存5件"的场景
  - 小商店的商品如果有规格（如不同包装的柴火），也需要SKU支持
- **修改建议**：
  1. 在数据字典中增加 **SKU（Stock Keeping Unit）** 实体：`product_id, spec_values(JSON), price, inventory`
  2. 或在PRD中明确"本期不支持SKU，每个规格创建独立商品"——但这会导致管理成本增加

### 问题6：购物车与票类商品的边界不清

- **位置**：PRD 第3.5.5节 购物车
- **问题描述**：
  - PRD定义"营位票/活动票/租赁不通过购物车，直接下单"——这个限制在产品层面合理，但需要技术上明确：
    - 如果用户想同时买2种不同规格的柴火 + 1箱啤酒，是1个订单还是3个订单？
    - 小商店商品和周边商品能否混合放入同一购物车？（它们的收入确认流程不同）
    - 如果混合购买，一个订单中部分商品即时到账、部分需确认收货后到账，如何处理？
- **修改建议**：
  1. 建议小商店和周边商品使用独立购物车（或在同一购物车中按品类分组结算）
  2. 明确"一个购物车→一个订单"还是"一个购物车→按品类拆分多个订单"
  3. 建议采用"拆单"策略：用户统一结算，系统按品类自动拆分为多个子订单，各自独立走收入确认流程

### 问题7：微信支付"模拟支付"方案不明确

- **位置**：PRD 第1.3节 项目范围
- **问题描述**：
  - PRD说"预留接口，暂不接入商户号，使用模拟支付流程"，但未定义：
    - 模拟支付的前端交互是什么？（直接点击"确认支付"跳过？展示一个假的支付页面？）
    - 模拟支付是否需要支持"支付失败"场景的测试？
    - 模拟支付的订单状态流转是否和真实支付一致？
    - 何时切换到真实支付？切换时需要改动多大？
    - 模拟退款如何处理？
- **修改建议**：
  1. 明确模拟支付的交互方案：建议前端展示支付确认按钮 + 模拟支付成功/失败的选择
  2. 明确后端的支付接口设计要求：采用策略模式，通过配置切换模拟/真实支付，确保接入真实支付时改动最小化
  3. 定义模拟支付的完整流程（包含回调）
  4. 退款的模拟方案

## 四、秒杀扩容设计建议（重点）

### 4.1 当前问题

PRD第5.1节性能要求中定义了"秒杀场景支持500+ QPS"，存在以下严重问题：

#### 4.1.1 500 QPS指标严重不足

- **场景分析**：假设一个万圣节活动有200个限量营位，活动预热期已有2000+用户关注。开票瞬间：
  - 2000用户同时点击"立即抢购"，瞬间QPS约2000-5000
  - 每个用户在前3秒内可能重复点击3-5次，有效QPS可达5000-15000
  - 加上页面刷新、倒计时轮询、库存查询等请求，总QPS可能达到10000-30000
- **结论**：500 QPS仅能支持极小规模的秒杀，随着品牌发展用户增长，很快会成为瓶颈
- **建议**：将秒杀QPS指标调整为"初期支持2000 QPS，架构设计支持水平扩展至50000 QPS"

#### 4.1.2 缺少秒杀的完整技术方案描述

当前PRD仅提到"使用Redis预扣库存，防止超卖"，这只是秒杀方案的冰山一角。一个完整的秒杀系统需要从前端到后端的全链路设计。

#### 4.1.3 缺少降级和容错方案

秒杀瞬间如果系统过载，当前PRD没有定义：
- 是否允许排队？排队的用户体验是什么？
- 系统过载时是否降级？降级策略是什么？
- 支付环节超时后的库存释放时间是多少？

### 4.2 建议补充的非功能需求

以下内容建议补充到PRD第5节"非功能需求"中：

#### 4.2.1 秒杀性能指标（修订）

### 4.3 建议补充的技术预埋点

以下扩容预埋点必须在架构设计阶段考虑，建议PRD在非功能需求中明确提出，以指导架构设计：

#### 预埋点1：多级限流策略

- **前端限流**：
  - 倒计时结束前按钮置灰，防止提前请求
  - 点击"立即抢购"后按钮置灰3秒，防止重复提交
  - 前端添加请求节流（throttle），限制1秒内最多1次请求
- **网关限流**：
  - 基于令牌桶/漏桶算法的API网关限流
  - 按用户维度限流（同一用户1秒内最多1次秒杀请求）
  - 按IP维度限流（同一IP 10秒内最多5次请求）
- **服务端限流**：
  - 信号量/线程池隔离秒杀接口
  - 请求队列削峰填谷

#### 预埋点2：请求队列与异步化

- 秒杀请求进入消息队列（如Redis List或RabbitMQ/RocketMQ），异步处理
- 前端轮询或WebSocket获取抢购结果
- 好处：削峰填谷，保护后端数据库
- **PRD需明确**：用户提交秒杀请求后，是否接受"排队中"的中间状态？最长等待时间是多少？

#### 预埋点3：库存预热与多级缓存

- 秒杀开始前将库存加载到Redis
- 使用Redis Lua脚本保证原子性扣减
- 本地缓存（JVM/内存缓存）作为第一级，Redis作为第二级
- 库存归零后直接在内存层拦截，不穿透到Redis

#### 预埋点4：CDN与静态化

- 商品详情页、活动页面的静态内容使用CDN加速
- 秒杀期间仅"抢购按钮"和"库存状态"走动态接口，其余静态化
- 倒计时使用客户端本地时间 + 服务端时间校准，避免频繁请求服务端

#### 预埋点5：数据库读写分离

- 主库处理订单写入、库存扣减
- 从库处理商品查询、订单查询、报表统计
- 秒杀期间非核心查询请求走从库，降低主库压力
- **PRD需明确**：查询数据存在1-3秒延迟是否可接受？

#### 预埋点6：Redis集群与缓存高可用

- Redis采用Cluster模式或Sentinel模式，支持故障自动转移
- 库存数据采用多分片策略，避免单点热点
- Redis宕机时的降级方案：降级为数据库直接扣减（性能降低但不影响正确性）

#### 预埋点7：服务水平扩展

- 后端服务必须无状态化，支持水平扩展
- Session存储在Redis而非本地
- 文件上传使用对象存储（如OSS/COS）而非本地磁盘
- 定时任务使用分布式调度（如XXL-JOB），避免多实例重复执行

#### 预埋点8：降级与熔断策略

- **秒杀降级**：
  - 库存归零后立即返回"已售罄"，不再访问后端服务
  - 非核心服务（如积分计算、消息通知）异步化，秒杀期间可延迟处理
  - 商品详情页走缓存/CDN，秒杀期间不查数据库
- **熔断策略**：
  - 支付服务超时时，自动熔断并释放库存
  - 下游服务（如身份证校验、微信API）熔断时不影响核心下单流程
- **PRD需明确**：秒杀期间是否可以关闭部分非核心功能（如"查看订单列表"、"积分查询"）以保护系统？

#### 预埋点9：秒杀活动隔离

- 秒杀流量与日常流量隔离，使用独立的接口路径、服务实例（可选）
- 避免秒杀流量影响正在浏览商品、已入营用户的正常使用体验
- **PRD需明确**：秒杀期间其他功能（小商店购物、验票等）是否要保证不受影响？

#### 预埋点10：监控与告警

- 秒杀期间的实时监控：QPS、响应时间、错误率、库存余量
- 库存异常告警（超卖检测）
- 服务器资源告警（CPU/内存/网络）

### 建议5：优先级调整建议

- **当前问题**：
  1. "页面编辑（首页内容配置）"放在P2，但首页轮播图和推荐商品是上线必备，建议提到P1
  2. 消息通知功能缺失，建议至少在P1中包含"订单状态通知"和"支付超时提醒"
  3. "收入管理"放在P1，但如果P0就要上线支付（哪怕是模拟支付），收入流转就必须同步实现，建议提到P0
- **建议调整**：
  - P0 增加：收入管理基础流程、订单状态通知
  - P1 增加：首页内容配置（从P2提升）、消息通知体系
  - P2 增加：评价系统、分享裂变、多营地支持

### 建议6：补充用户协议和隐私政策的具体要求

- **位置**：PRD 第5.3节 合规要求
- **问题**：PRD只提到"首次使用需同意用户协议和隐私政策"，但缺少：
  - 隐私政策需列明收集的个人信息类型（身份证、手机号、位置等）

  - 需提供用户注销账号的功能
  - 需符合《个人信息保护法》要求
  - 微信小程序审核对隐私说明文件有强制要求
- **建议**：在合规要求中增加PIPL合规清单

### 建议7：电子票二维码的防截图盗用机制

- **位置**：PRD 第3.5.3节 验票系统
- **问题**：当前二维码防伪方案是"过期时间+一次性Token"，但存在以下风险：
  - 用户截图分享给他人，在Token过期前被他人使用
  - 年卡会员的"动态验证码"更新频率未定义
- **建议**：
  1. 二维码内容每30秒刷新一次（或更短）
  2. 年卡会员的动态验证码建议6位数字，每60秒刷新
  3. 验票时校验用户当前位置（可选，需用户授权）
  4. 明确"一张票只能验票一次"还是"一张多日票可以每天验票一次"

### 建议追加章节二：订单生命周期定义（新增 3.7 节）

```markdown
### 3.7 订单生命周期（P0）

**订单状态流转**：

| 状态 | 说明 | 可执行操作 |
|------|------|-----------|
| 待支付 | 订单已创建，等待支付 | 支付、取消订单 |
| 已支付/待使用 | 支付成功，等待验票/使用 | 申请退款、查看电子票 |
| 已验票/使用中 | 已验票入营或已拿到装备 | 查看订单详情（不可退款，特殊情况联系客服） |
| 已完成 | 服务结束（离营/活动结束/归还装备/确认收货） | 评价（P2） |
| 已取消 | 用户主动取消或支付超时 | 无 |
| 退款中 | 用户申请退款，等待处理 | 管理员审批 |
| 已退款 | 退款完成 | 无 |

**状态流转规则**：
- 待支付 →（支付成功）→ 已支付/待使用
- 待支付 →（超时未支付/用户取消）→ 已取消（释放库存）
- 已支付/待使用 →（验票）→ 已验票/使用中（收入从待确认转为可提现）
- 已支付/待使用 →（申请退款 + 在可退期限内）→ 退款中
- 已验票/使用中 →（服务结束/归还/确认收货）→ 已完成
- 退款中 →（管理员审批通过）→ 已退款（退款到微信，释放库存）
- 退款中 →（管理员审批拒绝）→ 已支付/待使用

**支付超时规则**：
- 普通订单：30分钟未支付自动取消
- 秒杀订单：5分钟未支付自动取消（释放库存供其他用户抢购）

**退款规则**：
- 管理员配置退款截止时间：入营日/活动日/租赁日前N天或N小时
- 超过截止时间不可申请退款
- 退款金额 = 支付金额（全额退款，不支持部分退款——部分退款可作为P2功能）
```

### 建议追加章节三：秒杀系统非功能需求（追加到第5节）

```markdown
### 5.4 秒杀系统专项要求

#### 5.4.1 性能指标

| 指标 | 初期要求 | 可扩展目标 | 说明 |
|------|---------|-----------|------|
| 秒杀接口QPS | 2,000 | 50,000（水平扩展） | 单次秒杀活动的峰值并发 |

| 订单创建TPS | 500 | 5,000（水平扩展） | 每秒成功创建的订单数 |
| 库存最终一致性 | 10秒内 | 5秒内 | Redis与数据库的同步延迟 |

#### 5.4.2 扩容预埋要求

为确保系统在用户增长后能平滑扩容，架构设计阶段必须满足以下预埋要求：

1. **前端防护**：
   - 抢购按钮防重复提交（点击后置灰N秒）
   - 前端请求节流（throttle），每秒最多发起1次抢购请求

2. **网关层**：
   - 支持按用户、IP维度的限流配置
   - 支持令牌桶/漏桶限流算法
   - 限流规则可运行时动态调整，无需重启

3. **服务层**：
   - 后端服务无状态化，支持水平扩展
   - Session和状态信息存储在Redis，不依赖本地内存
   - 秒杀请求支持异步队列处理（预留消息队列接入点）
   - 服务间调用支持熔断和降级

4. **数据层**：
   - 库存使用Redis缓存 + Lua原子脚本扣减
   - 数据库支持读写分离部署
   - 预留分库分表扩展接口（按营地或按时间）

5. **基础设施**：
   - 静态资源接入CDN
   - 支持容器化部署（Docker），方便快速水平扩展
   - 监控和告警体系：QPS、响应时间、错误率、库存余量实时可观测

6. **降级策略**：
   - 库存归零后前端直接展示"已售罄"，不再请求后端
   - 秒杀期间非核心功能（积分计算、统计分析）可延迟处理
   - 支付回调失败时自动重试，超过重试次数进入人工处理队列

#### 5.4.3 秒杀用户体验要求

| 环节 | 要求 |
```

```markdown
### 7.2 扩展性设计预留

#### 7.2.1 多营地扩展

虽然一期仅运营单一营地，但数据模型需预留多营地扩展能力：
- 所有核心业务实体（Product、Inventory、Order、Staff等）需包含 `site_id` 字段
- 一期 `site_id` 默认值为固定值，但字段和索引需提前创建
- 后台管理需预留营地切换入口（一期可隐藏）

#### 7.2.2 品类扩展

Product表使用"基础表+扩展表"模式：
- Product基础表：所有品类共有的字段（名称、类型、状态、价格区间、图片等）
- 各品类扩展表：存储品类特有的业务字段
- 新增品类时只需创建新的扩展表，不影响现有功能
```
