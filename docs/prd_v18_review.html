1. 总体结论

v1.8 PRD 已基于最新 v1.7 实现重新定位为“统一商品售卖与现场运营增强”。文档明确复用 v1.7 二维码、扫码归因、订单导出、退款结算和电子票核销底座，没有重复建设已完成能力。

rev4 已按用户最新要求完成关键决策收敛：全部需求纳入同一个 v1.8 生产版本一次上线；跨商品共享库存池作为正式能力进入模型与验收；微信群助手固定按企业微信群机器人实现，并与小程序智能客服共用知识库、审计与人工兜底规则。

第一轮架构初评为 7.4 / 10，NOT APPROVED，主要阻塞项是库存池、临时订单状态机、SKU 迁移、会员卡统一、日历规则优先级、核销日志、浏览统计性能和 AI/高危安全边界。rev4 已补充这些关键约束，并消除了分阶段/多次验收口径，达到生产上线级 PRD 审批标准。

5. APPROVED 后开发文档要求

  • 后端开发文档必须把库存池、临时订单、支付、退款、核销的一致性闭环写成可实现状态机。
  • 后端开发文档必须明确跨商品共享库存池的绑定、锁定、扣减、释放、审计和迁移策略。
  • Admin 开发文档必须明确商品编辑器分步配置、日历库存编辑、会员卡统一、临时订单、高危确认、知识库和企业微信群机器人配置页面与权限。
  • 小程序开发文档必须明确手机号授权、商品详情可售单元、订单确认、会员卡、员工核销和 AI 客服交互。
  • 企业微信群机器人设计必须作为前端/运营触点由对应 agent 评审，包括配置页、测试发送、失败告警和审计视图。
  • 三端评审必须验证旧 SKU、旧年卡/次卡、历史订单、v1.7 二维码/导出/退款结算不被破坏。
