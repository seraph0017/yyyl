## 一、整体评价与评分

### 总体评分：9.7 / 10（上一轮 9.0 / 10，+0.7）

PRD v1.2 在 v1.1 的基础上，针对上一轮最终评审中提出的 **5 个 Nice-to-have 问题全部进行了高质量修复**，修复深度超出预期。产品经理不仅解决了问题本身，还主动补充了大量计算示例、边界 Case 和数据结构预埋设计，展现了极强的技术理解力和产品严谨性。

**核心改进亮点：**

1. **滑动窗口算法模型**（Nice-to-have #1）：从模糊的"连续预定5天需中断2天"进化为精确的自然日连续性判定算法，6组计算示例覆盖正常/触发/恢复/非连续/跨段等场景，5个边界 Case 覆盖多日票/跨月/整单校验/跨营位等极端情况，退票后额度恢复规则也同步定义
2. **多日票退款分级方案**（Nice-to-have #2）：创造性地引入"P0整单退票 + 客服手动部分退票"的分级策略，既控制了P0开发范围，又覆盖了实际业务场景，OrderItem 按天拆分的数据结构为 P2 自助部分退票奠定了坚实基础
3. **动态定价优先级链**（Nice-to-have #3）：4级优先级链定义清晰（特定日期特殊价 > 自定义日期类型价 > 系统日期类型价 > 商品基础价），4组定价查找示例直观易懂，注意事项说明了两种定价维度的差异，消除了歧义
4. **父订单+子订单模型**（Nice-to-have #4）：用户视角一次支付、系统视角拆分流转的设计完美平衡了体验和业务复杂度，数据字典同步更新了 `parent_order_id` 字段
5. **13项二次确认操作清单**（Nice-to-have #5）：从无到有建立了完整的高危操作防误操作体系，按风险等级区分两种确认方式，覆盖商品/库存/退款/权限/定价/会员/系统等7大操作域

**未扣满分的原因（-0.3）：**

- 遗留 #6（SKU/规格体系）仅有骨架预留，缺少详细规格设计
- 遗留 #7（分布式定时任务方案）PRD 层面无需详细定义，但缺少对"定时任务需覆盖的场景清单"的显式列举
- 遗留 #8（微信订阅消息模板）缺少具体模板类型与申请策略的映射

这3项属于"技术方案层面的设计"而非"产品需求层面的定义"，PRD 不扣大分，但作为完美主义的追求方向仍值得记录。

## 二、Nice-to-have 修复验证

### Nice-to-have #1: 年卡会员"连续预定5天需中断2天"规则的算法精确性

- **状态**：✅ 已解决（高质量修复，超出预期）
- **评审意见**：
  - PRD 3.2.1节对"连续预定限制"进行了**全面重构**，引入"滑动窗口算法说明"专项章节
  - **精确定义**：
    - "连续"定义为自然日连续性（D(i+1) = D(i) + 1），消除了上一轮"周一→周三→周五算不算连续"的歧义
    - 限制规则公式化：下次可预定最早日期 = D_end + GAP_DAYS + 1
    - 参数可配置（MAX_CONSECUTIVE 默认5、GAP_DAYS 默认2）
  - **计算示例**：6组示例覆盖了：
    1. 正常连续（4天+1天=5天，未超限，✅）
    2. 触发中断（5天后第6天，❌）
    3. 中断期内（5天后第7天仍被拒，❌）
    4. 中断后恢复（5天+空2天后第8天，✅）
    5. 非连续不累计（日期不连续则重新计数，✅）
    6. 跨中断段累计（中断后重新开始计数5天后再次触发，❌）
  - **退票后额度恢复**：退票后对应日期从已预定序列中移除，连续天数实时重新计算，并给出了具体示例
  - **5个边界 Case**：
    1. 多日票覆盖的每一天均独立计入
    2. 跨月份自然日连续性正常判定
    3. 一次下单多天需整体校验（整单拒绝，不做部分截断）
    4. 不同营位类型的预定天数合并计算（防绕过）
    5. 与3.7节退票规则联动一致
  - **架构师评价**：这是本次修订中最令人满意的改进。算法定义精确到可直接翻译为代码，计算示例覆盖了我在上一轮提出的所有疑问场景，边界 Case 第4条（跨营位类型合并计算）更是产品经理主动发现的安全漏洞防护。

### Nice-to-have #2: 多日票退款的日期粒度

- **状态**：✅ 已解决（高质量修复，超出预期）
- **评审意见**：
  - PRD 3.7节新增"多日票退款专项规则（P0阶段）"章节，采用**分级策略**：
    - **用户自助**：P0阶段仅支持整单全额退票
    - **客服手动**：管理员在后台"客服管理→手动部分退票"中逐天操作
  - **手动退票流程**5步清晰：客服确认→管理员勾选天数→系统计算退款金额→管理员确认→执行退款并释放库存
  - **退款金额计算规则**精确：
    - 连定优惠按天均摊：每天退款金额 = 该天原价 × 折扣率
    - 退部分天后剩余天数**不追溯**调整价格（不补收差价）——这个决策非常务实，避免了退款后连锁价格重算的复杂性
  - **OrderItem 按天拆分存储要求**：
    - 明确要求多日票下单时按天拆分为独立 OrderItem 记录
    - 每条 OrderItem 包含 `date`（日期）、`unit_price`（原价）、`actual_price`（折后实付价）、`refund_status`（退款状态）
    - 明确说明此设计为 P2 自助部分退票预留数据基础
  - **后台管理联动**：3.4节客服管理模块增加了"手动部分退票"功能描述，与3.7节互相引用
  - **架构师评价**：分级策略是产品设计上的聪明决策——既控制了 P0 的开发复杂度（用户端不需要复杂的部分退票UI），又通过客服后台手动操作覆盖了实际业务场景。数据结构预埋到位，P2 升级为用户自助部分退票时改动量可控。

### Nice-to-have #3: 动态定价中"自定义日期类型"的优先级规则

- **状态**：✅ 已解决（高质量修复）
- **评审意见**：
  - PRD 3.1.1节新增"动态定价优先级链"专项章节，定义了4级定价优先级：

    | 优先级 | 定价来源 | 说明 |
    |--------|---------|------|
    | 1（最高） | 特定日期特殊价 | 针对单个商品+单个日期的精确设置 |
    | 2 | 自定义日期类型价 | 管理员将某日期标记为自定义类型后按类型取价 |
    | 3 | 系统日期类型价 | 按日期自然属性（工作日/周末）取价 |
    | 4（兜底） | 商品基础价 | Product表中的默认价格 |

  - **定价查找示例**：4组示例清晰展示了优先级命中逻辑：
    - 5月1日（周四）：有特殊价258元 → 命中优先级1
    - 4月30日（周三）：有自定义类型"节假日" → 命中优先级2 → 198元
    - 4月26日（周六）：自然日类型"周末" → 命中优先级3 → 168元
    - 4月23日（周三）：自然日类型"工作日" → 命中优先级3 → 128元
  - **注意事项**：明确区分了优先级1（单个商品+单个日期）和优先级2（全营地某日期类型覆盖）的含义差异
  - **架构师评价**：优先级链定义清晰，示例直观。"命中即停止"的匹配逻辑简洁高效，可直接映射为代码中的责任链或策略模式。特别是最后的注意事项区分了两种定价的作用域差异，避免了开发时的理解歧义。

### Nice-to-have #4: 购物车拆单后的支付体验

- **状态**：✅ 已解决（高质量修复）
- **评审意见**：
  - PRD 3.5.5节购物车的"支付体验（父订单+子订单模型）"完整定义了：
    - **用户视角**：一次结算、一次支付、一个支付金额
    - **系统视角**：父订单（Parent Order）统一支付 + 子订单（Sub Order）独立流转
    - **具体规则**4条：
      1. 前端统一订单确认页和支付金额
      2. 支付后用户可见父订单和子订单列表
      3. 子订单可独立退款/查物流，互不影响
      4. 父订单状态跟随子订单（全部完成→已完成；任一退款→部分退款）
  - **数据字典联动**：Order 实体增加了 `parent_order_id` 字段，说明"父订单仅用于统一支付和展示，子订单独立流转业务状态"
  - **架构师评价**：模型设计合理。父订单聚合支付、子订单独立流转是电商领域的成熟模式。状态跟随规则（特别是"部分退款"状态）的定义消除了后端开发的歧义。唯一的小遗憾是未提及"父订单的退款金额展示逻辑"（是否为所有子订单已退金额之和），但这属于实现细节，不影响 PRD 质量。

### Nice-to-have #5: 缺少管理员操作的二次确认机制

- **状态**：✅ 已解决（高质量修复，超出预期）
- **评审意见**：
  - PRD 3.4节新增"高危操作二次确认机制"完整章节，内容包括：
  - **13项需二次确认的操作清单**：覆盖7大操作域：
    - 商品管理（2项）：批量下架、删除有订单商品
    - 库存管理（2项）：批量调整库存、库存清零
    - 退款审批（2项）：大额退款、批量退款
    - 客服管理（1项）：手动部分退票
    - 权限管理（2项）：管理员角色变更、员工权限修改
    - 定价管理（1项）：批量修改价格
    - 会员管理（1项）：大额积分调整
    - 系统设置（2项）：退票规则修改、支付模式切换
  - **两种确认方式按风险等级区分**：
    - 高风险：弹窗确认 + 输入系统随机6位确认码（确保已阅读操作摘要）
    - 极高风险：弹窗确认 + 输入管理员密码（身份验证）
  - **通用规则**5条：
    1. 确认弹窗必须展示影响范围
    2. 错误3次锁定15分钟
    3. 所有高危操作记录到操作日志
    4. 管理员可配置是否启用（默认强制开启）
    5. 金额和数量阈值可配置
  - **验收标准联动**：3.4节验收标准同步增加了4条二次确认相关验收项
  - **架构师评价**：这是本次修订中**设计最完整**的新增章节。13项操作清单覆盖面广，风险等级分层合理，确认码机制（随机码+手动输入）比简单的"确认/取消"弹窗安全性高出很多。错误锁定机制和审计日志要求也体现了安全纵深思维。唯一建议：可考虑在极高风险操作中增加"操作冷静期"（如确认后延迟30秒执行，期间可撤销），但这属于锦上添花。

## 三、遗留事项改善检查

### 遗留 #6: SKU/规格体系的详细设计

- **状态**：⚠️ 部分解决（骨架已有，细节待补）
- **评审意见**：
  - PRD 7.1节数据字典中 `ProductExtShop` 扩展表包含 `sku_specs` 字段
  - `Inventory` 实体包含 `sku_id` 字段
  - `Cart` 实体包含 `sku_id` 字段
  - 7.2节库存粒度定义中小商店/周边商品的库存维度为 `product_id + sku_id`
  - **不足之处**：
    - 缺少独立的 `SKU` 实体定义（sku_id、product_id、spec_values、price、stock、status 等）
    - `sku_specs` 字段的数据格式未定义（是JSON？结构如何？例如 `[{"name":"颜色","values":["红","蓝"]},{"name":"尺码","values":["S","M","L"]}]`）
    - 规格组合的笛卡尔积生成逻辑、SKU 级定价覆盖商品基础价等规则未说明
  - **影响评估**：SKU 体系主要影响 P1（小商店）和 P2（周边商品），P0 阶段影响不大。但架构设计阶段需要完整定义 SKU 数据模型
  - **结论**：PRD 层面已预留了足够的扩展点（sku_id 字段散布在多个实体中），架构设计阶段补充 SKU 实体和规格规则即可

### 遗留 #7: 分布式定时任务方案

- **状态**：⚠️ 部分解决（需求已隐含，场景清单未显式列举）
- **评审意见**：
  - PRD 中已隐含定义了多个需要定时任务的场景：
    1. 订单支付超时自动取消（3.7节：普通30分钟/秒杀5分钟）
    2. 积分过期自动清零（3.2.2节：12个月有效期）
    3. 年卡会员到期提醒（3.6节：到期前30天/7天/1天通知）
    4. 行程提醒（3.6节：入营日前一天通知）
    5. 活动开票提醒（3.6节：开票前1小时/10分钟通知）
    6. 定时开票状态切换（3.5.4节：到时间自动开放购买）
    7. 装备超时未归还提醒（3.1.3节异常处理）
  - **不足之处**：
    - 这些定时任务场景分散在各章节中，未汇总为"定时任务需求清单"
    - 未明确定时任务的执行精度要求（如订单超时是秒级精度还是分钟级轮询？）
    - 未明确定时任务失败后的重试/告警策略
  - **影响评估**：定时任务方案属于技术架构层面的选型决策，PRD 不必规定具体技术方案（如 XXL-JOB 还是 Spring Scheduled），但应明确场景清单和精度要求
  - **结论**：建议在架构设计阶段汇总定时任务清单，定义执行精度和失败策略。PRD 层面的需求描述已足够支撑技术方案设计

### 遗留 #8: 微信订阅消息模板的申请与审核策略

- **状态**：⚠️ 部分解决（通知场景已定义，模板策略未细化）
- **评审意见**：
  - PRD 3.6节消息通知体系已定义了 10 种通知场景，覆盖了核心业务节点
  - 5.5节微信小程序平台约束中明确了"订阅消息需用户主动订阅，每次订阅仅授权一次推送"
  - 业务规则中指出了"在关键节点引导用户订阅"的策略
  - **不足之处**：
    - 未列出每种通知场景对应的微信订阅消息模板类型（一次性订阅 vs 长期订阅）
    - 未说明模板申请的时间节点（应在开发前提前申请，审核周期约1-3个工作日）
    - 未明确每种通知场景的"订阅引导时机"（如支付成功页引导订阅"退款通知+物流更新+行程提醒"三个模板）
    - 未提及微信对模板关键词的限制（每个模板最多5个关键词，需提前规划字段映射）
  - **影响评估**：此项属于运营策略和技术对接的交叉地带。微信订阅消息模板需要在微信公众平台申请，审核有一定周期，且模板关键词一旦选定不易更改。建议在架构设计/开发准备阶段尽早梳理
  - **结论**：PRD 层面的通知需求已足够清晰，模板申请策略建议在技术方案文档中以"微信订阅消息对接方案"专项章节细化

## 四、新发现的问题（如有）

经过对 PRD v1.2 的逐字审查，发现以下**微小问题**，均为 Informational 级别，不影响评分和进入架构设计阶段的结论。

### 新问题1: 父订单退款金额展示逻辑未定义

- **严重程度**：Informational
- **问题描述**：3.5.5节定义了父订单状态跟随子订单（"任一子订单退款→父订单显示'部分退款'"），但未定义父订单的退款金额如何展示
- **建议**：父订单的退款金额 = 所有已退子订单的退款金额之和。可在架构设计阶段明确

### 新问题2: 手动部分退票中"退款金额含已分摊优惠"的边界情况

- **严重程度**：Informational
- **问题描述**：3.7节多日票退款专项规则中定义了"每天退款金额 = 该天原价 × 折扣率"和"退部分天后剩余天数不追溯调整价格"。但有一个极端场景：如果用户预定3天享受连定优惠（8折），退掉2天后只剩1天，用户实际只付了1天的8折价格却享受了1天的服务——这对营地方可能有轻微的收入损失
- **建议**：这是业务策略决策，PRD 已明确"不追溯调整"，说明产品经理接受了这个代价以换取用户体验和系统简洁性。架构师认同此决策，不建议修改

### 新问题3: 章节编号跳过第6节

- **严重程度**：Informational（格式问题）
- **问题描述**：PRD 目录从第5节（非功能需求）直接跳到第7节（数据字典），缺少第6节。实际上优先级列表（P0/P1/P2）位于5.5节之后、7.1节之前，但未标注章节号
