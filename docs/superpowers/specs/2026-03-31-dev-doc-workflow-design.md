# 开发文档拆分与评审工作流设计

> **日期**：2026-03-31
> **状态**：已批准

---

## 背景

项目现有「需求→PRD→架构师评审→开发」工作流。PRD 通过架构师评审后直接进入开发，缺少从 PRD 到各端具体开发文档的拆分与验证环节。需要新增「架构师拆分开发文档 → 端开发 Agent 评审 → 迭代修改」流程。

## 设计方案

### 总流程

```
PRD APPROVED
    → 架构师 Agent 拆分三端开发文档
    → 三端开发 Agent 并行 review
    → 架构师根据反馈修改
    → 迭代直到三端均 ≥8.5 APPROVED
    → 进入开发
```

### 架构师拆分规则

架构师 Agent 读取 APPROVED 的 PRD，输出三份独立开发文档到 `docs/` 目录：

| 文件名格式 | 内容范围 | 目标读者 |
|-----------|---------|---------|
| `docs/v{版本}_server_dev.md` | 数据模型变更、API 接口定义（路径/参数/响应）、业务逻辑、Celery 任务、中间件变更 | 后端开发 Agent |
| `docs/v{版本}_admin_dev.md` | 页面/组件设计、路由变更、API 调用清单、状态管理、样式规范 | Admin 前端开发 Agent |
| `docs/v{版本}_miniapp_dev.md` | 页面/组件设计、pages.json 变更、API 调用清单、类型定义、样式规范 | 小程序开发 Agent |

每份文档须包含：
1. 概述与开发范围
2. 文件清单（新增/修改的文件路径）
3. 详细技术方案（含代码示例）
4. 与其他端的接口约定
5. 不在本端范围的内容说明

### 端开发 Agent 评审规则

三个端开发 Agent **并行** review 各自的开发文档。

评审维度与权重：

| 维度 | 权重 | 检查项 |
|------|------|--------|
| API 接口完整性 | 25% | PRD 中定义的接口是否全部覆盖、参数/响应字段是否完整 |
| 数据模型可执行性 | 20% | 字段类型合理、索引设计正确、与现有模型兼容 |
| 技术方案兼容性 | 25% | 是否与现有代码模式一致、依赖是否已存在、文件路径是否正确 |
| 业务规则严密性 | 15% | PRD 中的业务规则是否被完整翻译为技术实现 |
| 可开发性 | 15% | 技术方案是否足够清晰、能否直接根据文档编码 |

评审输出格式：
- 逐维度评分（1-10）+ 加权总分
- CRITICAL/HIGH 问题列表（必须修复）
- MEDIUM 建议（可选改进）
- 结论：APPROVED（≥8.5）/ NEEDS REVISION（<8.5）
- 评审报告输出到 `docs/v{版本}_{端}_review.md`

### 迭代规则

- 架构师根据三端反馈修改对应开发文档
- 端开发 Agent 复审确认
- 迭代直到三端 review 均 ≥8.5 且结论为 APPROVED

### CLAUDE.md 变更

将原步骤 5「进入开发」拆分为：
- 步骤 5：开发文档拆分与评审
- 步骤 6：端开发 Agent 评审
- 步骤 7：迭代修订（开发文档）
- 步骤 8：进入开发

新增「开发文档版本历史」表格跟踪各端文档状态。
