2. 统一商品管理与购买选项（P0）

2.1 功能描述

管理后台将“营位管理”和“商品管理”统一为“商品管理”。对运营人员而言，营位、活动、租赁、商品都作为商品进行创建、筛选、上下架、配置价格、配置库存、生成二维码和查看订单。底层继续复用现有 Product 基表、类型扩展表、SKU 与 Inventory，新增结构化选项和库存模式以承接复杂购买选项。

2.2 类型映射

后台显示类型内部 Product.type说明
营位daily_camping, event_camping日常营位和活动营位
活动daily_activity, special_activity日常活动和特定活动；特定活动不单独建菜单，在商品管理筛选查看
租赁rental装备租赁
商品shop, merchandise, insurance现场商品、邮寄商品、保险等

2.3 购买选项规则

规则项规则说明
选项组每个商品可配置 0-N 个选项组，例如规格、颜色、位置、时间、套餐类型。
选项值每个选项组包含 1-N 个选项值，可排序、启用/停用。
SKU 生成多选项商品按选项组合生成 SKU；无选项商品保留默认 SKU。
价格SKU 可设置独立价格；未设置时使用 Product.base_price。
库存模式independent 独立库存、shared_pool 共享库存池。
独立库存每个 SKU/日期/场次独立扣减库存，适用于气罐规格颜色、营位位置。
共享库存多个商品、SKU、活动场次或租赁设备可通过 InventoryPoolBinding 指向同一库存池，用户购买任一绑定售卖单元均扣同一池，适用于皮划艇活动名额、跨商品联票、同一批租赁装备等。
兼容旧数据旧 SKU.spec_values 可迁移或兼容读取；未迁移商品按默认 SKU 处理。

2.4 商品购买逻辑

商品场景购买规则
营位必须选择日期；按包含人数和入住晚数计算详细订单数。
活动可配置预定时间/场次；按人数或套票单位下单。
租赁可配置租赁日期/归还规则；可收押金；按件数下单。
现场商品只需选择 SKU 与数量，可直接购买或加入购物车，不要求地址。
邮寄商品必须选择收货地址；订单保留物流字段。
保险/搭配商品沿用 v1.5/v1.7 搭配与订单规则，不在本期扩展独立购买。

2.5 数据模型变更

表变更
Product新增 display_type 可选冗余字段、sale_mode 现场/邮寄/预约、cover_image_url、view_count 可选统计缓存。
ProductOptionGroup新增：id, site_id, product_id, name, code, sort_order, status。
ProductOptionValue新增：id, group_id, label, value, sort_order, status, extra_price_delta。
SKU新增/规范：option_value_ids, inventory_mode, inventory_pool_id(nullable), sort_order。
InventoryPool新增：id, site_id, pool_code, name, pool_type, total, available, locked, sold, status；作为跨商品共享库存事实源。
InventoryPoolBinding新增：id, site_id, inventory_pool_id, product_id, sku_id, activity_session_id, rental_asset_id, priority, status；显式声明哪些售卖单元共享同一库存池。
Inventory补唯一约束 (site_id, product_id, sku_id, date, time_slot)；新增 inventory_pool_id，绑定库存池时库存扣减以库存池为准。

2.7 验收标准

  • 后台商品管理可筛选商品、营位、活动、租赁。
  • 气罐可按规格+颜色生成独立库存 SKU。
  • 万圣节营位票可按位置生成独立库存 SKU。
  • 皮划艇活动、联票商品和租赁商品可跨商品共享同一库存池，并能追溯每个商品的占用明细。
  • 现场商品无需地址即可下单，邮寄商品必须选择地址。
  • 旧商品数据不丢失，旧营位入口可跳转到统一商品管理。

3. 库存日历与批量库存管理（P0）

保留营位日历用于日常订票，同时新增活动日历和租赁日历。管理员可在日历中查看指定日期的库存状态、订单列表、核销状态和销售情况，并支持按日期区间、星期规则批量关闭或调整库存。

日历覆盖商品主要用途
营位日历daily_camping, event_camping日常订票、关闭营位、批量调库存
活动日历daily_activity, special_activity活动场次容量、报名订单查看
租赁日历rental装备可租数量、日期库存管理
规则项规则说明
单日关闭可关闭某商品/SKU/库存池在某一天的库存，关闭后 C端不可下单。
星期规则可按日期范围 + 星期批量关闭，例如 2026-01-01 至 2026-02-01 每周六关闭。
库存调整支持增加/减少 total；available 自动按 total - locked - sold 校验，不能小于 0。
订单弹窗点击日历日期可查看当天订单：订单号、用户、手机号、商品/SKU、数量、核销状态、备注。
并发保护库存调整与下单锁库存必须使用行级锁；调整不得覆盖已售/锁定库存。
审计所有人工库存调整写入 InventoryLog，记录前后值、操作人、规则 ID 和备注。
方法路径说明
GET/api/v1/admin/inventory/calendar多类型库存日历查询
GET/api/v1/admin/inventory/day-orders查询某日订单弹窗
POST/api/v1/admin/inventory/adjust单日库存增减/关闭/开放
POST/api/v1/admin/inventory/batch-rules创建批量关闭/库存调整规则
GET/api/v1/admin/inventory/logs库存调整日志
  • 管理员可关闭 2026-01-01 至 2026-02-01 每周六的指定营位库存。
  • 点击日历日期能弹窗查看当天订单和核销状态。
  • 人工减少库存不得导致可用库存为负。
  • 活动和租赁也可按日期/场次查看和调整库存。

4. 微信手机号登录与订单用户信息（P0）

小程序用户登录必须完成微信手机号授权。订单与后台用户管理需要展示微信用户关联的系统用户、手机号和订单备注。当前 /api/v1/auth/phone-login 仍是 TODO，仅返回普通登录结果；v1.8 必须补真实微信 getPhoneNumber 流程。

规则项规则说明
登录流程小程序先 wx.login 获取 code，再通过手机号授权按钮获取 phone_code，调用后端 phone-login。
绑定手机号后端调用微信手机号接口，成功后写入 User.phone，并返回带手机号的 token 用户信息。
强制授权未绑定手机号的用户可浏览商品，但下单、会员卡购买、客服提问、核销展示前必须授权手机号。
隐私普通 admin 默认展示脱敏手机号；拥有订单导出/用户管理权限才可查看完整手机号。
订单字段说明
订单号Order.order_no
用户系统 User.id/nickname/openid 脱敏展示
手机号User.phone 或订单快照手机号，默认脱敏
订单金额total/actual/refunded/settled
订单数商品类=数量；营位=人数 x 晚数；活动=人数/套票 x 场次
订单状态服务状态 + 支付状态 + 退款状态 + 核销状态分开展示
订单备注下单时用户备注 + 后台内部备注分离
  • 新用户首次下单前必须完成手机号授权。
  • 授权成功后 User.phone 落库，订单列表可按手机号搜索。
  • 管理后台订单详情展示订单号、用户、手机号、金额、订单数、核销状态、备注。

5. 会员卡统一与二维码购买（P0）

后台不再将年卡和次卡作为两个割裂入口，而统一为“会员卡”。开卡时可选择：无限次数仅限制有效期，或限制次数且限制有效期。不同营位/商品可配置不同消耗次数或消耗选项，后台可手动调整次数和有效期，会员卡可生成小程序码购买。

现有能力v1.8 策略
AnnualCardConfig/AnnualCard映射为 usage_mode=unlimited 的会员卡
TimesCardConfig/TimesCard映射为 usage_mode=limited_times 的会员卡
激活码保留，作为会员卡发放方式之一
次数消耗规则升级为会员卡消耗规则，可按商品/SKU/营位区域配置
规则项规则说明
卡配置卡名、价格、有效期、使用模式、总次数、适用商品、状态、购买二维码。
无限次数有效期内可用，仍受每日限额、连续天数、适用商品限制。
限制次数每次下单按消耗规则扣减，次数不足不可下单。
消耗规则可按商品、SKU、营位区域设置 consume_count；未配置默认 1 次。
后台调整super_admin/admin 可调整剩余次数、有效期、状态；必须记录调整原因和日志。
二维码购买复用 v1.7 小程序码，目标类型新增/复用 membership_card。
  • 后台以“会员卡”统一管理年卡和次卡。
  • 创建会员卡时可选择无限次数或限制次数。
  • 不同营位可配置不同消耗次数。
  • 后台可调整用户会员卡剩余次数和有效期，且有日志。

6. 核销与手机端商家查看增强（P0）

购买营位、活动、租赁、现场商品后均生成可核销凭证。员工通过小程序手机端扫码核销，并可查看当天订单、待核销列表、核销历史和订单详情。现有 /pages/staff 与 Ticket QR 流程保留并增强。

规则项规则说明
凭证生成支付成功后按订单项生成 Ticket/核销凭证；商品类可按数量生成一张多次数凭证或多张凭证，由商品配置决定。
动态二维码沿用 30 秒刷新机制，降低截图复用风险。
核销权限staff/admin/super_admin 可核销；必须校验员工 site_id 与票券 site_id。
幂等同一二维码重复扫码返回“已核销”和首次核销信息，不重复扣次数。
核销日志每次成功/失败核销均记录 TicketVerifyLog，含员工、时间、设备、结果、失败原因。
手机端查看员工可查看今天订单、待核销、已核销、用户手机号脱敏、备注。
年卡/会员卡会员卡核销保留验证码二次确认或等价用户确认机制。

10. 高危操作二次确认加固（上线必选）

设置超级管理员六位数字操作密码，用于提现，并作为未来最高权限需求（批量导出、批量删除、高危退款、临时订单大额收款等）的二次确认。当前确认接口必须加固，不能保留“确认码非空即通过”或简单 SHA256 校验。

规则项规则说明
密码格式六位数字，仅 super_admin 可设置/重置。
哈希使用 bcrypt/argon2，不使用明文、MD5、SHA256 单轮哈希。
错误锁定连续错误 5 次锁定 15 分钟，可配置。
确认 token验证成功后返回短 TTL token，绑定 admin_id、action、site_id、request_hash。
动作绑定token 只能用于本次动作，不可复用到其他高危接口。
审计成功/失败均写 OperationLog，失败不记录明文密码。
高危动作提现、完整手机号导出、订单导出、批量删除、高危退款、会员卡大额调整、临时订单大额收款。

11. 智能客服与企业微信群助手（上线必选）

管理员上传露营知识、营地须知、活动规则、退款规则、交通指引等资料，系统构建知识库。用户可在小程序智能客服中打字提问，系统基于知识库回答并给出来源。企业微信群助手通过企业微信群机器人能力接入，和小程序智能客服同步上线并复用同一知识库。

功能v1.8 全量上线范围
小程序智能客服支持文本问答、来源引用、人工客服兜底、回答反馈和会话日志。
企业微信群助手通过企业微信群机器人 webhook 或企业微信应用消息接入，支持群内常见问题自动回复、无法回答转人工提示、回复审计。
大量知识提交支持后台富文本/Markdown/文件上传；文件解析类型至少覆盖 txt/md/pdf/docx。
敏感问题涉及价格、退款、天气、订单状态时优先调用系统 API 或提示人工确认。
  • 每个 site_id 独立知识库，不跨营地检索。
  • 上传文件执行大小、类型、病毒/脚本风险校验。
  • 回答必须限制在知识库和可调用系统信息内，不编造价格、库存、退款承诺。
  • Prompt injection 内容不得改变系统身份或泄露后台数据。

16. 架构修补闭环

16.1 库存池、临时订单与支付一致性

主题规则
库存事实源库存事实以 InventoryPool / InventorySlot 或增强后的 Inventory 为准，库存池用于表达同商品或跨商品共享库存，商品/SKU/活动场次/租赁设备通过 InventoryPoolBinding 显式绑定库存池，不自行维护可售事实。
不变量任一库存槽必须满足 available + locked + sold = total，且三个数量均不得小于 0。
锁库存时机创建正式订单或临时订单确认支付前锁库存，生成 stock_hold_token；未支付过期、取消或支付失败必须释放锁定库存。
锁定时长普通订单默认 30 分钟，临时现场收款默认 15 分钟，秒杀订单沿用现有秒杀支付超时配置。
幂等键订单预览、临时订单确认、支付回调、退款释放库存均必须携带或生成幂等键，重复请求不能重复扣减或释放库存。
支付成功微信支付成功通知幂等确认库存：locked 减少、sold 增加，并生成票券/核销凭证及资金 pending 入账。
退款/取消沿用 v1.7 退款状态机；只有明确 release_inventory=true 的退款或取消才回补 available，且通过库存日志记录。

16.2 商品选项与旧 SKU 兼容迁移

阶段规则
兼容读取旧商品继续读取 SKU.spec_values 与 ProductExtShop.spec_definitions；新商品优先读取 ProductOptionGroup/ProductOptionValue。
历史订单历史订单永远以 OrderItem 快照金额、SKU 名称、规格快照为准，不因后台修改 SKU/选项而重算。
渐进迁移提供后台迁移任务，将旧 JSON 规格转换为结构化选项；迁移前后 SKU id 尽量保持不变。
双写限制迁移期允许服务层兼容读，不允许 Admin 同时编辑旧 JSON 与新结构，避免双写冲突。
回滚迁移任务必须记录迁移批次和原始 JSON 备份，支持按商品回滚到迁移前状态。

16.3 日历规则优先级

优先级规则层说明
1站点默认规则默认营业日、节假日、基础日期类型。
2商品规则商品可售周期、停售时间、活动有效日期。
3库存池规则同商品或跨商品共享库存池的日期/场次容量规则。
4批量日历规则按日期范围和星期生成的开关/库存调整规则。
5单日覆盖手工单日关闭、开放或指定库存，优先级最高。

冲突处理原则：关闭优先于开放；单日覆盖优先于批量规则；不得将 total 调整到低于 sold + locked；价格规则与库存规则分离，但日历接口需同时返回最终价格与最终可售库存。

16.4 会员卡权益引擎与迁移

主题规则
权益计划MemberCardPlan 表达卡产品，包含 usage_mode、validity_days、total_times、适用商品和销售状态。
权益实例MemberCardInstance 表达用户持有卡，记录 start/end、remaining_times、status、来源订单。
权益流水MemberCardUsageLedger 记录每次消耗、返还、人工调整，作为次数余额事实源。
旧数据兼容年卡映射为 unlimited，次数卡映射为 limited_times；迁移前可通过统一服务聚合展示。
退款返还会员卡抵扣订单退款时，按原订单消耗流水返还权益；现金退款与权益返还分开记录。

16.5 核销、统计、AI 与高危安全边界

领域必须约束
核销新增 TicketVerificationLog，成功、失败、重复核销、验证码失败均记录；员工核销必须校验 admin/user staff 身份与 site_id。
浏览统计浏览事件先写 Redis/队列，按分钟或小时批量落库；UV 去重不得导出 openid 明文。
AI 客服回答必须基于已发布知识库或系统 API；无来源时转人工，不得编造库存、价格、退款结论。
高危确认操作密码使用 bcrypt/argon2，失败锁定，确认 token 绑定动作、请求摘要、site_id 和短 TTL。
