## 1. 项目概述

### 1.1 项目背景

"一月一露"是一家户外露营地品牌，提供营位预定、装备租赁、户外活动、商品售卖等综合服务。目前通过微信群、电话等方式管理预约和售卖，效率低、易出错、无法规模化。需要一个微信小程序来实现线上化运营，提升用户体验和运营效率。

### 1.2 项目目标

1. **用户端**：为露营爱好者提供一站式的营位预定、装备租赁、活动报名、商品购买服务
2. **运营端**：为营地运营方提供商品管理、订单管理、会员管理、财务管理、数据统计等后台能力
3. **效率提升**：实现全流程线上化，包括开票→预定→支付→验票→入营→离开→收入确认

### 1.3 项目范围

| 范围 | 说明 |
|------|------|
| 包含 | 微信小程序用户端（C端）、员工端（小程序内页面）、后台管理API、**独立Web管理后台（含Dashboard数据看板）** |
| 不包含 | — |
| 微信支付 | 预留接口，暂不接入商户号，使用模拟支付流程（详见下方） |

**模拟支付方案**：

| 环节 | 方案说明 |
|------|----------|
| 前端交互 | 点击"支付"后展示模拟支付页面，提供"模拟支付成功"和"模拟支付失败"两个按钮，方便测试两种场景 |
| 后端架构 | 采用策略模式（PaymentStrategy接口），通过配置项切换模拟/真实支付实现。接入真实支付时只需新增 WechatPayStrategy 实现类，无需修改业务代码 |
| 状态流转 | 模拟支付的订单状态流转与真实支付完全一致（待支付→已支付→...），确保业务逻辑不受支付方式影响 |
| 回调模拟 | 模拟支付成功后，后端自动触发支付回调逻辑，模拟微信回调通知 |
| 退款模拟 | 退款时直接更新订单状态为"已退款"，模拟退款到账 |
| 切换成本 | 接入真实支付时只需：① 配置微信商户号和密钥 ② 实现 WechatPayStrategy ③ 修改配置切换策略。预计改动量 ≤ 2个文件 |

## 2. 目标用户

### 2.1 用户画像

| 用户角色 | 描述 | 核心诉求 |
|----------|------|----------|
| **露营爱好者（C端用户）** | 25-45岁，城市白领/家庭用户，喜爱户外活动，使用微信为主 | 方便快捷地预定营位、参加活动、购买装备和商品 |
| **年卡会员** | 高频露营用户，每月至少一次露营 | 享受会员权益，免费/优惠预定，专属活动 |
| **营地管理员** | 营地老板/运营负责人 | 管理商品、订单、财务、库存，查看数据统计 |
| **营地员工** | 前台/验票/仓管人员 | 扫码验票、查看订单、处理退款 |

### 2.2 核心使用场景

1. **周末露营**：用户周三晚上打开小程序，选择本周六-周日的营位，选择有电有木平台的位置，填写身份信息，签署免责声明，支付下单
2. **活动秒杀**：万圣节活动票4月1日20:00开售，用户提前进入页面等待倒计时结束，快速下单
3. **装备租赁**：用户预定营位后，浏览装备租赁分类，选择帐篷和桌椅套装，支付租金和押金，到营地后自行取用，归还后管理员检查无损一键退还押金
4. **现场自助购物**：用户需要柴火，打开小程序小商店，选择柴火加入购物车，自助付款后去取货
5. **员工验票**：用户到达营地，出示电子票二维码，员工用小程序扫码验票确认入营

## 3. 功能需求

### 3.1 七大售卖品类

#### 3.1.1 日常露营票订票（P0）

**功能描述**：用户可在非活动时间预定营位票，包括过夜露营票和单日票。

**用户故事**：
- 作为露营爱好者，我想选择日期预定营位，以便安排周末露营
- 作为露营爱好者，我想根据是否有电、是否有木平台等条件筛选营位，以便找到适合的位置
- 作为露营爱好者，我想连续预定多天并享受折扣，以节省费用

**业务规则**：

| 规则项 | 规则说明 |
|--------|----------|
| 票种类型 | 过夜露营票（两天一夜、三天两夜…）、单日票 |
| 预定模式 | ① 按位置预定（单一营位，孤品，选中即占用）② 按人数预定（自由营位，限定总数，加人票） |
| 加人票规则 | ① 加人票需关联原营位订单，仅在原订单有效期间内可购买 ② 加人票需独立填写身份信息（姓名、身份证、手机号）③ 加人票享受原订单的日期类型定价（工作日/周末/节假日），但不独立享受连定优惠 ④ 加人票生成独立电子票，验票时关联原订单 ⑤ 加人票退票规则与原票一致，退加人票不影响原订单 ⑥ 人数优惠按原订单总人数（含加人票）统一计算 |
| 动态定价 | 工作日/周末/节假日价格不同。管理员可自定义日期的类型（如将某个周三定义为节假日价格）。管理员也可对特定日期设置特殊价格（覆盖日期类型价格）。示例：工作日128元，周末168元，节假日198元。**定价优先级链见下方说明** |
| 定时开票 | 管理员可设置开票时间。如：4月1日20:00开放5月1日至5月30日的票 |
| 连定优惠 | 规则可配置，具体规则如下：① **连日优惠**：连续预定N天（N≥2），全部天数享受X折。示例：连续预定2天，第1天工作日128元×0.8=102.4元，第2天周末168元×0.8=134.4元，每天独立计价后各自打折。② **多人优惠**：同一订单人数≥5人时，全部人享受X折。③ **优惠互斥规则**：多种优惠（连定/多人/会员折扣/积分抵扣）不叠加，系统自动计算各方案取最优价展示给用户。④ 年卡会员免费预定的营位不适用连定优惠（已免费）。⑤ 管理员可在后台配置连定天数阈值、人数阈值、折扣比例 |
| 退票规则 | 管理员可配置退票截止时间：提前N天或XX小时可退票 |
| 身份登记 | 管理员可按商品配置登记模式（必填/选填/不需要）和字段组合（系统内置字段+自定义字段），详见3.5.1节 |
| 免责声明 | 购票前必须签署免责声明，记录签署时间和签署人信息，具有法律效力 |
| 库存管理 | 每种营位有独立库存，管理员可批量开启/关闭库存 |
| 详情页面 | 每种票有独立的详情页面，包含：商品介绍、图片轮播、营位属性标签（有电/无电、阳光/阴凉、有木平台/无木平台）|

**动态定价优先级链**：

系统在查找某个商品在某个日期的价格时，按以下优先级从高到低依次匹配，命中即停止：

| 优先级 | 定价来源 | 说明 | 示例 |
|--------|---------|------|------|
| **1（最高）** | 特定日期特殊价（PricingRule.custom_date） | 管理员针对某个商品在某个**具体日期**设置的固定价格，优先级最高 | 营位A在5月1日设为特殊价 258元 |
| **2** | 自定义日期类型价（DateTypeConfig + PricingRule.date_type） | 管理员先将某个日期标记为自定义类型（如把周三标记为"节假日"），再按该类型的定价规则取价 | 4月30日（周三）被标记为"节假日"→ 取节假日价 198元 |
| **3** | 系统日期类型价（自然日历类型 + PricingRule.date_type） | 按日期自然属性（周一至周五=工作日，周六周日=周末）匹配对应的类型定价 | 普通周六 → 取周末价 168元 |
| **4（兜底）** | 商品基础价 | Product表中的默认价格，当以上三级均未配置时使用 | 取商品默认价 128元 |

**定价查找示例**：

假设营位A的定价配置为：工作日128元，周末168元，节假日198元。

| 日期 | 是否有特殊价 | 是否有自定义类型 | 自然日类型 | 最终价格 | 命中规则 |
|------|------------|----------------|-----------|---------|---------|
| 5月1日（周四） | 有，258元 | 有，标记为"节假日" | 工作日 | **258元** | 优先级1：特定日期特殊价 |
| 4月30日（周三） | 无 | 有，标记为"节假日" | 工作日 | **198元** | 优先级2：自定义类型→节假日价 |
| 4月26日（周六） | 无 | 无 | 周末 | **168元** | 优先级3：系统日期类型→周末价 |
| 4月23日（周三） | 无 | 无 | 工作日 | **128元** | 优先级3：系统日期类型→工作日价 |

> **注意**：优先级1的"特定日期特殊价"是针对**单个商品+单个日期**的精确设置；优先级2的"自定义日期类型"是针对**全营地某个日期**的类型覆盖，影响所有商品。两者含义不同，互不冲突。

**异常处理**：
- 库存不足：提示"该日期已约满"，建议选择其他日期
- 开票时间未到：显示倒计时，不可下单
- 退票超时：提示"已超过退票时间，无法退票"
- 身份证号格式校验失败：提示重新输入

**验收标准**：
- [ ] 用户可按日期、营位类型、属性筛选并预定
- [ ] 不同日期类型显示不同价格
- [ ] 连定优惠自动计算折扣
- [ ] 定时开票前显示倒计时，到时间自动开放
- [ ] 退票规则按配置执行
- [ ] 身份信息可保存和复用
- [ ] 免责声明签署记录完整

#### 3.1.7 营地周边售卖（P2）

**功能描述**：类似电商模式，用户可购买衣服、帽子、天幕等周边商品，支持邮寄发货。

**用户故事**：
- 作为露营爱好者，我想购买营地品牌的天幕
- 作为管理员，我想导出所有待发货订单的收货地址

**业务规则**：

| 规则项 | 规则说明 |
|--------|----------|
| 购买方式 | 按数量购买，限定总库存 |

**功能描述**：用户可付费购买年卡会员，享受专属权益。

**用户故事**：
- 作为高频露营用户，我想购买年卡，每次露营不用再付营位费
- 作为管理员，我想灵活配置年卡的权益和限制

**业务规则**：

| 规则项 | 规则说明 |
|--------|----------|
| 购买方式 | 支付固定费用购买，一年内有效 |
| 营位权益 | 管理员可配置：某种营位一年无限次免费预定，某种营位一年限制N次预定 |
| 专属活动 | 特定活动仅年卡会员可报名 |
| 预定频率限制 | ① **每日限额**（详见下方说明）② 连续预定限制（滑动窗口模型，详见下方说明） |
| 可配置项 | 有效期、会员价格、各营位预约次数均可由管理员调节 |
| 身份登记 | 年卡购买时必须登记身份证号和姓名（年卡级别固定必填，不受商品级配置影响） |
| 验票展示 | 验票时显示年卡会员身份标识。年卡会员需通过**验证码防转借机制**完成验票（员工扫码→用户手机弹出验证码输入界面→用户输入员工端展示的动态验证码→验票完成），详见3.5.3节 |

**每日限额规则说明**：

> 每日限额与连续预定限制（滑动窗口）是**两套独立的限制规则**，两者同时生效，任一不满足即拒绝预定。

| 预定模式 | 每日限额 | 默认值 | 可配置 | 说明 |
|---------|---------|--------|--------|------|
| 按位置预定（营位票/孤品） | 每天最多预定N张 | 1张 | ✅ 管理员可调 | 同一用户同一天最多预定N个不同的按位置营位 |
| 按人数预定（自由票） | 每天最多预定N份 | 2份 | ✅ 管理员可调 | 同一用户同一天最多预定N份自由票 |

**"份"的定义**：
- "按人数预定每天2份"中的**"份"= 订单份数**，一个订单不论包含几人，算1份
- **加人票不计入每日限额**：加人票属于原订单的附属票，不独立占用每日预定额度

**"不可每日重复预定"规则**：
- 同一天对**同一个商品**不能重复预定（如：已预定了3月15日的"A区1号营位"，不可再次预定3月15日的"A区1号营位"）
- 但同一天可以预定**不同商品**（如：已预定了3月15日的"A区1号营位"，还可以预定3月15日的"B区2号营位"，只要未超过每日限额）

**连续预定限制——滑动窗口算法说明**：

> 以下参数均可由管理员在后台配置，默认值为：最大连续天数 MAX_CONSECUTIVE = 5，中断天数 GAP_DAYS = 2。

1. **"连续"精确定义**：按预定日期的**自然日连续性**判定。若会员已有预定日期序列 D1, D2, …, Dn，当所有相邻日期满足 D(i+1) = D(i) + 1 时，视为连续。
2. **限制规则**：连续入住达到 MAX_CONSECUTIVE 个自然日后，下一次预定的入住日期必须 ≥ 最后一个连续日 + GAP_DAYS + 1。即：若连续预定的最后一天为 D_end，则下次可预定的最早日期为 D_end + GAP_DAYS + 1（中间至少空出 GAP_DAYS 个完整自然日）。
3. **计算示例**（MAX_CONSECUTIVE=5, GAP_DAYS=2）：

| 场景 | 已预定日期 | 新预定日期 | 是否允许 | 原因 |
|------|-----------|-----------|---------|------|
| 正常连续 | 3月1日-3月4日（4天） | 3月5日 | ✅ 允许 | 连续仅5天，未超限 |
| 触发中断 | 3月1日-3月5日（5天） | 3月6日 | ❌ 拒绝 | 已连续5天，3月6日在中断期内 |
| 触发中断 | 3月1日-3月5日（5天） | 3月7日 | ❌ 拒绝 | 3月7日仍在中断期内（需空出3月6-7两天） |
| 中断后恢复 | 3月1日-3月5日（5天） | 3月8日 | ✅ 允许 | 3月8日 = 3月5日 + 2 + 1，中断期已满 |
| 非连续不累计 | 3月1日-3月3日 + 3月5日 | 3月6日 | ✅ 允许 | 3月3日→3月5日不连续，3月5-6日仅连续2天 |
| 跨中断段累计 | 3月1日-3月5日 + 3月8日-3月12日 | 3月13日 | ❌ 拒绝 | 3月8日重新计数，3月8日-12日已连续5天，需再次中断 |

4. **退票后额度恢复**：用户退票成功后，对应日期从已预定序列中移除，连续天数实时重新计算。例如：已预定3月1日-5日（5天触发中断），退掉3月5日后，已预定变为3月1日-4日（4天），3月5日及之后可正常预定。
5. **边界Case**：
   - 多日票覆盖的每一天均独立计入连续天数序列
   - 跨月份的自然日连续性正常判定（如3月31日→4月1日视为连续）
   - 一次下单预定多天时需整体校验：若新增预定会导致连续天数超过 MAX_CONSECUTIVE，则整单拒绝，不做部分截断
   - 不同营位类型的预定天数**合并计算**（防止通过切换营位类型绕过限制）

**异常处理**：
- 会员过期：提示续费，过期后权益停止
- 达到预定上限：提示"今日预定已达上限"或"连续预定已达上限，请等待X天后再预定"

**验收标准**：
- [ ] 年卡购买和生效流程完整
- [ ] 各项权益按配置正确执行
- [ ] 预定限制规则正确执行
- [ ] 验票时正确展示会员身份和验证码

#### 3.2.3 营位次数卡（P1）

**功能描述**：用户通过输入秘钥（激活码/兑换码）领取次数卡，使用次数卡可免费预定指定营位（扣减次数代替付费）。

**用户故事**：
- 作为用户，我想用收到的兑换码领取次数卡，免费预定营位
- 作为管理员，我想批量生成兑换码分发给合作方/活动参与者
- 作为管理员，我想为不同档次的营位设置不同的次数消耗

**业务规则**：

| 规则项 | 规则说明 |
|--------|----------|
| 领取方式 | 用户在小程序内输入秘钥（激活码/兑换码）领取次数卡。一个秘钥仅可使用一次 |
| 秘钥管理 | 管理员在后台批量生成秘钥，每个秘钥对应一个次数卡配置（次数、有效期、可用营位范围）。秘钥格式：16位字母数字组合，支持批量导出 |
| 次数机制 | 次数卡有总次数（total_times），每次预定消耗对应次数。不同营位消耗不同次数（如豪华营位消耗3次，普通营位消耗1次），管理员可在后台配置每种营位的消耗次数 |
| 次数调整 | 管理员可在后台手动调整某张次数卡的剩余次数（增加或减少） |
| 有效期 | 次数卡有有效期（start_date - end_date），过期后剩余次数失效，不可再使用 |
| 可用范围 | 每张次数卡配置可适用的营位/商品范围，不在范围内的营位不可使用次数卡预定 |
| 预定免费 | 次数卡预定的营位不收费（扣次数即可），不走支付流程，直接生成已支付订单 |
| 与年卡共存 | 次数卡可与年卡会员共存（同一用户可同时拥有），预定时用户可选择使用年卡权益还是次数卡 |
| 次数不足 | 次数不足时提示"次数不足，请联系客服"，不可下单 |
| 退票规则 | 次数卡预定退票后，返还对应消耗的次数（返还至原次数卡，但不超过总次数） |
| 积分 | 次数卡预定不产生消费金额，因此不累积积分 |
| 身份登记 | 次数卡预定同样遵循该商品的身份登记配置（引用3.5.1节），登记模式和字段组合与普通付费预定一致 |
| 验票规则 | 次数卡用户验票流程与普通用户相同（扫码即完成），不需要验证码防转借（因次数有限，转借风险低于年卡） |

**次数卡与年卡共存规则**：

> 次数卡与年卡是两套独立的权益体系，互不影响、互不消耗。预定时用户需明确选择使用哪一种权益。

| 规则项 | 规则说明 |
|--------|----------|
| 独立于年卡权益体系 | 次数卡预定**不受年卡每日限额和滑动窗口限制**，两者的限额/频率限制各自独立 |
| 次数卡独立限额 | 次数卡有独立的每日预定限额（管理员可配置，默认无限制） |
| 不消耗年卡额度 | 次数卡预定**不消耗年卡的"限制N次"额度**，反之亦然 |
| 权益选择交互 | 如果用户同时满足年卡和次数卡的使用条件，系统弹窗让用户选择使用哪种权益（默认推荐**即将过期的权益**优先使用） |
| 同一订单不可混合 | 同一订单**不可混合使用**年卡权益和次数卡，必须选择其一 |
| 多张次数卡持有 | 一个用户可同时持有多张次数卡。预定时展示**有效次数卡列表**（按过期时间升序排列），用户选择用哪张卡扣减次数 |
| 日期类型无关 | 次数消耗与日期类型无关：不论工作日/周末/节假日，同一营位消耗**固定次数**（以TimesConsumptionRule配置为准） |
| 不可转让 | 次数卡绑定个人，**不可转让**给其他用户 |
| 连定优惠不适用 | 次数卡预定**不适用连定优惠**（因为不涉及金额，无折扣计算基础） |

**applicable_products与TimesConsumptionRule关系**：

| 规则项 | 规则说明 |
|--------|----------|
| 白名单机制 | applicable_products作为**白名单**，只有列表中的营位/商品才可使用次数卡预定 |
| 默认消耗规则 | 如果营位在白名单中但**未配置TimesConsumptionRule**，默认消耗1次 |
| 规则约束 | TimesConsumptionRule中的product_id**必须在applicable_products范围内**，后台配置时需校验 |

**次数消耗配置示例**：

| 营位类型 | 消耗次数 | 说明 |
|---------|---------|------|
| 普通营位（无电无平台） | 1次 | 基础消耗 |
| 标准营位（有电/有平台） | 2次 | 中等消耗 |
| 豪华营位（有电+有平台+景观位） | 3次 | 高消耗 |

> 以上为示例，管理员可自由配置每种营位的消耗次数。

**异常处理**：
- 秘钥已使用：提示"该兑换码已被使用"
- 秘钥不存在：提示"无效的兑换码，请确认后重试"
- 次数卡已过期：提示"次数卡已过期，剩余次数已失效"
- 次数不足：提示"次数不足，当前剩余X次，预定该营位需要Y次"

**验收标准**：
- [ ] 用户可通过秘钥正确领取次数卡
- [ ] 不同营位按配置的消耗次数正确扣减

- [ ] 管理员可调整次数卡剩余次数

### 3.3 收入管理（P0）

**功能描述**：多种不同的收入确认模式，确保资金安全。

**业务规则**：

| 收入类型 | 确认流程 | 退款规则 |
|----------|----------|----------|
| 日常/活动露营票 | 支付→进入待确认账户→验票入营→转入可提现账户 | 验票前可退→从待确认扣除；验票后联系客服→从可提现扣除 |
| 装备租赁-租赁费 | 支付→进入待确认账户→验票拿装备→转入可提现账户 | 拿装备前可退→从待确认扣除 |
| 装备租赁-押金 | 支付→进入押金专户（不计入收入）→归还后退押金 / 损坏扣除后退剩余 | 归还后一键退还；损坏时管理员选择损坏等级→系统计算扣除金额→通知用户→退剩余押金 |
| 日常/特定活动票 | 支付→进入待确认账户→验票参加活动→转入可提现账户 | 活动前可退→从待确认扣除 |
| 小商店售卖 | 支付→立即进入可提现账户 | 联系客服退款→从可提现扣除 |
| 周边商品售卖 | 支付→进入待确认账户→用户确认收货→转入可提现账户 | 收货前可退→从待确认扣除 |
| 年卡会员费 | 支付→立即进入可提现账户 | 购买后7天内可无条件退款→从可提现扣除 |
| 积分兑换订单 | 无资金流转，仅扣减积分和库存 | 取消订单→返还积分、恢复库存 |
| 次数卡预定订单 | 无资金流转，仅扣减次数和库存 | 退票→返还对应消耗次数、恢复库存 |

**押金管理补充规则**：
- 押金单独核算，不计入营业收入
- 管理员退还押金时支持全额退还和部分扣除两种方式

- 扣除的押金记入"设备维护收入"独立科目
- 损坏赔偿比例表由管理员在后台配置

**验收标准**：
- [ ] 7大品类收入流转路径各自正确
- [ ] 待确认和可提现金额显示正确
- [ ] 入营验票后自动转入可提现
- [ ] 押金单独核算，不计入营业收入
- [ ] 押金部分扣除流程完整（拍照→评估→扣除→通知→退款）
- [ ] 退款时从正确的账户扣除
- [ ] 年卡会员费7天内退款正确执行
- [ ] 积分兑换订单无资金流转但库存正确扣减

**年卡会员验票流程（防转借验证）**：

> 目的：防止年卡会员将二维码截图发给他人代替入营。验证码仅出现在用户自己的手机上，必须本人在场才能完成验票。

| 步骤 | 操作方 | 具体描述 |
|------|--------|---------|
| 步骤1 | 用户 | 用户打开小程序电子票页面，页面显示二维码（二维码每30秒刷新） |
| 步骤2 | 员工 | 员工使用小程序扫码功能扫描用户的二维码 |
| 步骤3 | 系统→用户 | 扫码成功后，**用户的手机上实时弹出验证码输入界面**，显示提示"请输入验证码完成验票"。同时员工端显示"等待用户输入验证码" |
| 步骤4 | 用户 | 用户在自己手机上的验证码输入界面中，输入**员工端展示的动态验证码**（6位数字） |
| 步骤5 | 系统 | 系统校验验证码正确后，验票完成，双方页面同步显示"验票成功" |

**验证码技术规则**：

| 规则项 | 说明 |
|--------|------|
| 验证码生成 | 由服务端实时生成6位随机数字验证码 |
| 验证码展示 | 验证码展示在**员工端**（扫码后员工屏幕上显示验证码），需要用户看到员工的屏幕后手动输入 |
| 有效期 | 每个验证码有效期60秒，超时自动失效，需员工重新扫码 |
| 一次性 | 验证码使用后立即失效，不可重复使用 |
| 通信方式 | 优先使用**长轮询（Long Polling）**实现员工端扫码后用户端实时弹出输入界面，避免使用WebSocket（受小程序WebSocket连接数限制） |
| 失败处理 | 验证码输入错误3次后，本次验票流程终止，需重新扫码 |
| 非年卡用户 | 普通用户（非年卡会员）验票流程不变，扫码即完成验票，无需验证码 |
| 用户小程序在后台/被杀掉 | 员工端提示"等待用户打开小程序"，用户重新打开小程序电子票页面后自动恢复验票流程（通过轮询检测待验证状态） |
| 多日票/年卡每日验票 | 多日票年卡会员**每天验票都需要完整的验证码流程**（不简化），确保每次入营均为本人到场 |

| 已支付/待使用 | 支付成功，等待验票/使用 | 申请退款、查看电子票 |
| 已验票/使用中 | 已验票入营或已拿到装备 | 查看订单详情（不可退款，特殊情况联系客服） |
| 已完成 | 服务结束（离营/活动结束/归还装备/确认收货） | 评价（P2） |
| 已取消 | 用户主动取消或支付超时 | 无 |
| 退款中 | 用户申请退款，等待处理 | 管理员审批 |
| 已退款 | 退款完成 | 无 |

**状态流转规则**：

```
待支付 →（支付成功）→ 已支付/待使用
待支付 →（超时未支付/用户取消）→ 已取消（释放库存）
已支付/待使用 →（验票）→ 已验票/使用中（收入从待确认转为可提现）
已支付/待使用 →（申请退款 + 在可退期限内）→ 退款中
已验票/使用中 →（服务结束/归还/确认收货）→ 已完成
退款中 →（管理员审批通过）→ 已退款（退款到微信，释放库存）
退款中 →（管理员审批拒绝）→ 已支付/待使用
```

**支付超时规则**：

| 订单类型 | 超时时间 | 超时处理 |
|---------|---------|---------|
| 普通订单 | 30分钟未支付 | 自动取消，释放库存 |
| 秒杀订单 | 5分钟未支付 | 自动取消，释放库存供其他用户抢购 |

**退款规则**：
- 管理员配置退款截止时间：入营日/活动日/租赁日前N天或N小时
- 超过截止时间不可申请退款
- 退款金额 = 支付金额（全额退款）
- 部分退票（如3天预定只退第3天）作为P2功能预留

**多日票退款专项规则（P0阶段）**：

| 规则项 | 规则说明 |
|--------|----------|
| 用户自助退票 | P0阶段多日票仅支持**整单全额退票**（所有天数一起退），不支持用户自助退单独某一天 |
| 客服手动部分退票 | 用户如需只退部分天数，需联系客服，由管理员在后台"客服管理→手动部分退票"中操作（详见3.4节客服管理模块） |
| 手动退票流程 | ① 客服确认用户诉求和要退的日期 → ② 管理员在后台选择该订单，勾选要退的天数对应的OrderItem → ③ 系统计算退款金额（按各天实际支付价格，含已分摊的优惠） → ④ 管理员确认退款金额 → ⑤ 执行退款并释放对应日期库存 |
| 退款金额计算 | 多日票的连定优惠按天均摊：每天退款金额 = 该天原价 × 折扣率。退部分天后，剩余天数**不追溯**调整价格（即不因退票后不满足连定条件而补收差价） |
| 退票后连续天数影响 | 年卡会员退票后，对应日期从连续预定序列中移除，连续天数实时重新计算（与3.2.1节规则一致） |

**OrderItem按天拆分存储要求**：
- 多日票下单时，系统必须将每一天拆分为独立的OrderItem记录存储（例如预定3天则生成3条OrderItem，每条记录独立的date、price）
- 每条OrderItem包含字段：order_id、product_id、**date（该天日期）**、quantity、**unit_price（该天原价）**、**actual_price（折后实付价）**、identity_id、parent_order_item_id
- 此设计为P2阶段用户自助部分退票功能预留数据基础，P0阶段即需按此结构存储

**验收标准**：
- [ ] 订单状态按流转规则正确流转
- [ ] 普通订单30分钟超时自动取消
- [ ] 秒杀订单5分钟超时自动取消
- [ ] 超时取消后库存正确释放
- [ ] 退款审批流程正确执行

### 5.4 秒杀系统专项要求

#### 5.4.1 性能指标

| 指标 | 初期要求 | 可扩展目标 | 说明 |
|------|---------|-----------|------|
| 秒杀接口QPS | 2,000 | 50,000（水平扩展） | 单次秒杀活动的峰值并发 |
| 秒杀接口P99响应时间 | ≤ 200ms | ≤ 100ms | 从用户点击到返回结果 |
| 库存扣减延迟 | ≤ 50ms | ≤ 20ms | Redis原子操作 |
| 订单创建TPS | 500 | 5,000（水平扩展） | 每秒成功创建的订单数 |
| 库存最终一致性 | 10秒内 | 5秒内 | Redis与数据库的同步延迟 |

#### 5.4.2 扩容预埋要求

为确保系统在用户增长后能平滑扩容，架构设计阶段必须满足以下预埋要求：

1. **前端防护**：
   - 抢购按钮防重复提交（点击后置灰N秒）
   - 前端请求节流（throttle），每秒最多发起1次抢购请求
   - 倒计时基于客户端本地时间 + 服务端校准，不轮询服务端

2. **网关层**：
   - 支持按用户、IP维度的限流配置
   - 支持令牌桶/漏桶限流算法
   - 限流规则可运行时动态调整，无需重启

3. **服务层**：
   - 后端服务无状态化，支持水平扩展
   - Session和状态信息存储在Redis，不依赖本地内存
   - 秒杀请求支持异步队列处理（预留消息队列接入点）
   - 服务间调用支持熔断和降级

4. **数据层**：
   - 库存使用Redis缓存 + Lua原子脚本扣减
   - 数据库支持读写分离部署
   - 预留分库分表扩展接口（按营地或按时间）

5. **基础设施**：
   - 静态资源接入CDN
   - 支持容器化部署（Docker），方便快速水平扩展
   - 监控和告警体系：QPS、响应时间、错误率、库存余量实时可观测

6. **降级策略**：
   - 库存归零后前端直接展示"已售罄"，不再请求后端
   - 秒杀期间非核心功能（积分计算、统计分析）可延迟处理
   - 支付回调失败时自动重试，超过重试次数进入人工处理队列

7. **秒杀流量隔离**：
   - 秒杀流量与日常流量隔离，使用独立的接口路径
   - 秒杀期间其他功能（小商店购物、验票、浏览等）保证不受影响

#### 5.4.3 秒杀用户体验要求

| 环节 | 要求 |
|------|------|
| 秒杀前 | 展示倒计时（精确到秒），提前展示商品详情和价格，支持提前填写身份信息和签署免责声明 |
| 秒杀中 | 下单流程≤3步，身份信息自动填充，免责声明提前签署 |
| 抢购成功 | 立即跳转支付页面，5分钟内完成支付，超时释放库存 |
| 抢购失败 | 展示"已售罄"，推荐其他可预定日期或活动 |
| 排队（扩展） | 若启用队列模式，展示"排队中，前方还有N人"，最长等待30秒，超时自动退出 |

### 7.2 库存粒度定义

不同品类的库存管理粒度不同：

| 品类 | 库存维度 | 说明 |
|------|---------|------|
| 日常露营票（按位置） | product_id + date | 每个营位每天库存=1（孤品） |
| 日常露营票（按人数） | product_id + date | 自由营位按每天人数上限管理 |
| 活动露营票 | product_id | 固定日期，总库存限制 |
| 装备租赁 | product_id + date | 同一装备不同日期可被不同人租赁 |
| 日常/特定活动 | product_id + date + time_slot | 按场次时间段管理 |
| 小商店/周边商品 | product_id + sku_id | 不按日期，仅按商品/规格管理 |

### 7.3 库存状态流转

```
可用(available) →（用户下单）→ 锁定(locked)
锁定(locked) →（支付成功）→ 已售(sold)
锁定(locked) →（支付超时/用户取消）→ 释放回可用(available)
已售(sold) →（退款完成）→ 释放回可用(available)
已售(sold) →（装备归还）→ 释放回可用(available)
```

- 支付超时释放时间：秒杀场景5分钟，普通场景30分钟
- 入库操作：管理员增加total和available
- 出库操作：系统售出减少available；管理员手动出库减少total和available

### 7.4 扩展性设计预留

#### 7.4.1 多营地扩展

虽然一期仅运营单一营地，但数据模型需预留多营地扩展能力：
- 所有核心业务实体（Product、Inventory、Order、Staff等）包含 `site_id` 字段
- 一期 `site_id` 默认值为固定值，但字段和索引需提前创建
- 后台管理需预留营地切换入口（一期可隐藏）

#### 7.4.2 品类扩展

Product表使用"基础表+扩展表"模式：
- Product基础表：所有品类共有字段（名称、类型、状态、价格区间、图片等）
- 各品类扩展表：存储品类特有的业务字段（如露营票的营位属性、装备租赁的押金等）
- 新增品类时只需创建新的扩展表，不影响现有功能
